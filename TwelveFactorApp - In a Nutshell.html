    üìñ Overview

    intro overview methodology

    In the modern era, software is commonly delivered as a service: called web apps, or software-as-a-service. The Twelve-Factor App is a methodology for building software-as-a-service apps. Written by Adam Wiggins, it synthesises experience and observations on a wide variety of SaaS apps at Heroku, paying particular attention to the dynamics of organic growth of an app over time, the dynamics of collaboration between developers working on the codebase, and avoiding the cost of software erosion.

    Goals of the Methodology

    A twelve-factor app is designed to:

    • Use declarative formats for setup automation, to minimize time and cost for new developers joining the project
    • Have a clean contract with the underlying operating system, offering maximum portability between execution environments
    • Be suitable for deployment on modern cloud platforms, obviating the need for servers and systems administration
    • Minimize divergence between development and production, enabling continuous deployment for maximum agility
    • Scale up without significant changes to tooling, architecture, or development practices

    Who Should Read This?

    Any developer building applications which run as a service. Ops engineers who deploy or manage such applications. The methodology applies to apps written in any programming language and using any combination of backing services (database, queue, memory cache, etc.).

    I. Codebase üèóÔ∏è

    factor 1 git vcs

    Core Principle

    "One codebase tracked in revision control, many deploys."

    A twelve-factor app is always tracked in a version control system, such as Git, Mercurial, or Subversion. A copy of the revision tracking database is known as a code repository, often shortened to repo.

    A codebase is any single repo (in a centralised revision control system like Subversion), or any set of repos who share a root commit (in a decentralised revision control system like Git).

    One-to-One Correlation: Codebase ‚Üî App

    • If there are multiple codebases, it's not an app ‚Äî it's a distributed system. Each component in a distributed system is an app, and each can individually comply with twelve-factor.
    • Multiple apps sharing the same code is a violation of twelve-factor. The solution is to factor shared code into libraries which can be included through the dependency manager.

    One Codebase, Many Deploys

    There is only one codebase per app, but there will be many deploys. A deploy is a running instance of the app ‚Äî typically a production site, one or more staging sites, and every developer's local environment (each of which also qualifies as a deploy).

    The codebase is the same across all deploys, although different versions may be active in each deploy. For example, a developer has some commits not yet deployed to staging; staging has some commits not yet deployed to production ‚Äî but they all share the same codebase.

    ‚úÖ Do

    • One repository per app / microservice
    • Use tags or commit hashes to identify specific releases
    • Factor shared code into versioned libraries via a dependency manager
    • Every developer's local environment counts as a deploy ‚Äî treat it that way

    ‚ùå Don't

    • Maintain multiple repos for what is logically a single app
    • Copy-paste code between apps ‚Äî use shared libraries instead
    • Let multiple apps share a codebase ‚Äî split them properly

    II. Dependencies üì¶

    factor 2 packages isolation

    Core Principle

    "Explicitly declare and isolate dependencies."

    Most programming languages offer a packaging system for distributing support libraries. Libraries can be installed system-wide ("site packages") or scoped into the directory containing the app ("vendoring" or "bundling").

    A twelve-factor app never relies on the implicit existence of system-wide packages. It declares all dependencies, completely and exactly, via a dependency declaration manifest. Furthermore, it uses a dependency isolation tool during execution to ensure that no implicit dependencies "leak in" from the surrounding system. The full and explicit dependency specification is applied uniformly to both production and development.

    Declaration + Isolation Must Always Be Used Together

    Only one or the other is not sufficient to satisfy twelve-factor:

    Benefits of Explicit Dependencies

    • Onboarding simplicity: A new developer can check out the app onto their machine with only the language runtime and dependency manager as prerequisites ‚Äî then run a deterministic build command to set everything up
    • No environment assumptions: No reliance on the specific environment of the production server
    • Reproducible builds: Same manifest always produces the same dependency tree

    System Tools Must Also Be Vendored

    Twelve-factor apps do not rely on the implicit existence of any system tools. Examples include shelling out to ImageMagick or curl. While these tools may exist on many systems, there is no guarantee they will exist on all systems where the app may run in the future, or whether the version found will be compatible. If the app needs to shell out to a system tool, that tool should be vendored into the app.

    üí° Pro Tip: Commit lock files (package-lock.json, Pipfile.lock, Gemfile.lock) to VCS to ensure reproducible builds across all environments and CI pipelines.

    III. Config ‚öôÔ∏è

    factor 3 environment configuration

    Core Principle

    "Store config in the environment."

    An app's config is everything that is likely to vary between deploys (staging, production, developer environments, etc). This includes:

    • Resource handles to the database, Memcached, and other backing services
    • Credentials to external services such as Amazon S3 or Twitter
    • Per-deploy values such as the canonical hostname for the deploy

    The Litmus Test

    üîç Litmus Test: Whether an app has all config correctly factored out of the code is whether the codebase could be made open source at any moment, without compromising any credentials.

    What Config Is Not

    This definition of "config" does not include internal application config, such as config/routes.rb in Rails, or how code modules are connected in Spring. This type of config does not vary between deploys, and so is best done in the code.

    Why Not Config Files?

    Using config files not checked into revision control (e.g., config/database.yml in Rails) is an improvement over constants in code, but still has weaknesses:

    • Easy to mistakenly check in a config file to the repo
    • Config files tend to be scattered in different places and different formats, making it hard to see and manage all config in one place
    • These formats tend to be language- or framework-specific

    Environment Variables: The 12-Factor Way

    The twelve-factor app stores config in environment variables (env vars). Env vars are:

    • Easy to change between deploys without changing any code
    • Little chance of being checked into the code repo accidentally
    • A language- and OS-agnostic standard

    ‚ö†Ô∏è The "Environments" Anti-Pattern

    Some apps batch config into named groups called environments (e.g., development, test, production in Rails). This method does not scale cleanly: as more deploys are created, new environment names are necessary (staging, qa, joes-staging), resulting in a combinatorial explosion of config which makes managing deploys very brittle.

    In a twelve-factor app, env vars are granular controls, each fully orthogonal to other env vars. They are never grouped together as "environments", but instead are independently managed for each deploy. This model scales up smoothly as the app expands into more deploys over its lifetime.

    ‚úÖ Do

    • Use environment variables for all config that varies by deploy
    • Use a secret manager (e.g., Azure Key Vault) for sensitive values
    • Test that your code would be safe to open-source ‚Äî no embedded credentials

    ‚ùå Don't

    • Commit secrets or connection strings to source control
    • Group config as named "environments" (dev/staging/prod) inside the codebase
    • Use language- or framework-specific config file formats to store per-deploy config
    # .env.example (safe to commit ‚Äî no real secrets)

    IV. Backing Services üîå

    factor 4 services resources

    Core Principle

    "Treat backing services as attached resources."

    A backing service is any service the app consumes over the network as part of its normal operation. Examples include datastores (such as MySQL or CouchDB), messaging/queueing systems (such as RabbitMQ or Beanstalkd), SMTP services for outbound email (such as Postfix), and caching systems (such as Memcached).

    Local vs. Third-Party: No Distinction

    Backing services like the database are traditionally managed by the same systems administrators who deploy the app's runtime. In addition to these locally-managed services, the app may also have services provided and managed by third parties. Examples include SMTP services (such as Postmark), metrics-gathering services (such as New Relic or Loggly), binary asset services (such as Amazon S3), and even API-accessible consumer services (such as Twitter, Google Maps, or Last.fm).

    The code for a twelve-factor app makes no distinction between local and third-party services. To the app, both are attached resources, accessed via a URL or other locator/credentials stored in the config.

    Resources Can Be Attached and Detached

    Each distinct backing service is a resource. The twelve-factor app treats these databases as attached resources, which indicates their loose coupling to the deploy they are attached to.

    Resources can be attached to and detached from deploys at will. For example, if the app's database is misbehaving due to a hardware issue, the administrator might spin up a new database server restored from a recent backup. The current production database could be detached, and the new database attached ‚Äî all without any code changes. Only the resource handle in the config needs to change.

    üí° Key Insight: A local MySQL database and one managed by Amazon RDS are both just backing services to the app. Swapping one for the other requires only a config change ‚Äî zero code changes.

    V. Build, Release, Run üöÄ

    factor 5 ci/cd deployment

    Core Principle

    "Strictly separate build and run stages."

    A codebase is transformed into a (non-development) deploy through three stages:

    The Three Stages Defined

    • The build stage is a transform which converts a code repo into an executable bundle known as a build. Using a version of the code at a commit specified by the deployment process, the build stage fetches vendor dependencies and compiles binaries and assets.
    • The release stage takes the build produced by the build stage and combines it with the deploy's current config. The resulting release contains both the build and the config and is ready for immediate execution in the execution environment.
    • The run stage (also known as "runtime") runs the app in the execution environment, by launching some set of the app's processes against a selected release.

    Stage Responsibilities

    StageInputOutputWho InitiatesImmutable?
    BuildSource code + deps at a specific commitBuild artifact (Docker image, JAR, binary)Developer (on deploy)‚úÖ Yes
    ReleaseBuild artifact + deploy configVersioned, timestamped releaseDeployment tooling‚úÖ Yes ‚Äî append-only ledger

    Release Management & Immutability

    The twelve-factor app uses strict separation between the build, release, and run stages. It is impossible to make changes to the code at runtime, since there is no way to propagate those changes back to the build stage.

    Every release should always have a unique release ID, such as a timestamp of the release (e.g., 2011-04-06-20:32:17) or an incrementing number (e.g., v100). Releases are an append-only ledger ‚Äî a release cannot be mutated once it is created. Any change must create a new release.

    Deployment tools typically offer release management tools, notably the ability to roll back to a previous release. For example, Capistrano stores releases in a subdirectory named releases, where the current release is a symlink to the current release directory ‚Äî making rollback straightforward.

    Why the Run Stage Must Be Simple

    Builds are initiated by the app's developers whenever new code is deployed. Runtime execution, by contrast, can happen automatically ‚Äî such as a server reboot, or a crashed process being restarted by the process manager. Therefore, the run stage should be kept to as few moving parts as possible, since problems that prevent an app from running can cause it to break in the middle of the night when no developers are on hand. The build stage can be more complex, since errors are always in the foreground for a developer who is driving the deploy.

    CI/CD Pipeline Mapping

    • Build: docker build, dotnet publish, npm run build, mvn package
    • Release: Tag Docker image with release ID, inject deploy-specific env config, Helm chart release
    • Run: kubectl apply, docker run, Azure Container Apps deploy, platform restart
    üîë Rule: A release is an immutable, append-only artefact. Any desired change must create a new release. This makes rollbacks trivially easy ‚Äî just reactivate a previous release.

    VI. Processes ‚ö°

    factor 6 stateless processes

    Core Principle

    "Execute the app as one or more stateless processes."

    Twelve-factor processes are stateless and share nothing. Any data that needs to persist must be stored in a stateful backing service (e.g., a database or cache). The process handles requests and discards all in-memory state between requests.

    Stateless vs. Stateful Processes

    What Must NOT be in Process Memory

    • User sessions ‚Äî store in Redis, database, or a distributed cache
    • Uploaded files ‚Äî store in object storage (Azure Blob, S3)
    • Cached computation results needed across requests ‚Äî use a cache service
    üí° Benefit: Stateless processes can be started, stopped, or duplicated at any time with no side effects ‚Äî enabling seamless horizontal scaling and fault tolerance.

    VII. Port Binding üåê

    factor 7 http self-contained

    Core Principle

    "Export services via port binding."

    The twelve-factor app is completely self-contained and does not rely on runtime injection of a webserver into the execution environment to create a web-facing service. The web app exports HTTP as a service by binding to a port, and listening to requests coming in on that port.

    üîç Key Contrast: Web apps are sometimes executed inside a webserver container ‚Äî PHP apps might run inside Apache HTTPD, or Java apps inside Tomcat. The twelve-factor app breaks this dependency entirely by embedding the server as a library via dependency declaration.

    Example: Port Binding in Node.js

    Beyond HTTP

    HTTP is not the only service that can be exported by port binding. Nearly any kind of server software can be run via a process binding to a port and awaiting incoming requests:

    • ejabberd ‚Äî speaks XMPP protocol
    • Redis ‚Äî speaks the Redis protocol
    • Any TCP/UDP-based protocol server
    üí° App-to-App Backing Service: The port-binding approach means that one app can become the backing service for another app ‚Äî by providing the URL to the backing app as a resource handle in the config for the consuming app.

    VIII. Concurrency üîÑ

    factor 8 scaling process model

    Core Principle

    "Scale out via the process model."

    In the twelve-factor app, processes are a first class citizen. Processes take strong cues from the Unix process model for running service daemons. Using this model, the developer can architect their app to handle diverse workloads by assigning each type of work to a process type. For example, HTTP requests may be handled by a web process, and long-running background tasks handled by a worker process.

    Process Formation

    The array of process types and number of processes of each type is known as the process formation.

    Process TypeDescriptionExample
    webHandles HTTP requestsgunicorn app:app
    workerBackground processingQueue consumers, ETL jobs
    clockScheduled tasksCron-like periodic jobs
    schedulerTask dispatchingCelery beat, Azure Functions timer

    Horizontal vs. Vertical Scaling

    Individual processes can still handle their own internal multiplexing via threads inside the runtime VM, or the async/evented model found in tools such as EventMachine, Twisted, or Node.js. But an individual VM can only grow so large (vertical scale), so the application must also be able to span multiple processes running on multiple physical machines.

    The process model truly shines when it comes time to scale out. The share-nothing, horizontally partitionable nature of twelve-factor app processes means that adding more concurrency is a simple and reliable operation.

    Process Manager

    Twelve-factor app processes should never daemonize or write PID files. Instead, rely on the operating system's process manager to manage output streams, respond to crashed processes, and handle user-initiated restarts and shutdowns:

    • Production: systemd, or a distributed process manager on a cloud platform
    • Development: Foreman, Docker Compose
    • Kubernetes: Pod lifecycle management, Horizontal Pod Autoscaler (HPA / KEDA)
    üí° Design for Horizontal Scale: Never assume a request will be handled by the same process or machine as the previous request. Process state must never be held in memory ‚Äî use backing services (cache, database) for shared state.

    IX. Disposability üí™

    factor 9 startup shutdown

    Core Principle

    "Maximize robustness with fast startup and graceful shutdown."

    The twelve-factor app's processes are disposable, meaning they can be started or stopped at a moment's notice. This facilitates fast elastic scaling, rapid deployment of code or config changes, and robustness of production deploys.

    Fast Startup

    Processes should strive to minimize startup time. Ideally, a process takes a few seconds from the time the launch command is executed until the process is up and ready to receive requests or jobs. Short startup time provides more agility for the release process and scaling up; and it aids robustness, because the process manager can more easily move processes to new physical machines when warranted.

    Graceful Shutdown

    Processes shut down gracefully when they receive a SIGTERM signal from the process manager.

    Process TypeGraceful Shutdown Behaviour
    Web process Cease listening on the service port (refuse new requests) ‚Üí allow current requests to finish ‚Üí exit. HTTP requests should be short (a few seconds); long-polling clients should seamlessly reconnect when the connection is lost.
    Worker process Return the current job to the work queue. On RabbitMQ: send a NACK. On Beanstalkd: job is returned to the queue automatically when worker disconnects.
    Lock-based jobs Release the lock on the job record. All jobs must be reentrant ‚Äî wrap results in a transaction, or make the operation idempotent.

    Robustness Against Sudden Death

    Processes should also be robust against sudden death in the case of a failure in the underlying hardware. A recommended approach is use of a robust queueing backend, such as Beanstalkd, that returns jobs to the queue when clients disconnect or time out.

    üí° Crash-Only Design: A twelve-factor app is architected to handle unexpected, non-graceful terminations. Crash-only design takes this concept to its logical conclusion ‚Äî if your app handles a crash correctly, graceful shutdown becomes a bonus, not a requirement.

    Graceful Shutdown in .NET

    var builder = WebApplication.CreateBuilder(args);
var app = builder.Build();

// Register cancellation token on SIGTERM
var lifetime = app.Services.GetRequiredService<IHostApplicationLifetime>();
lifetime.ApplicationStopping.Register(() =>
{
    // Graceful shutdown: drain connections, flush buffers, return jobs to queue
    Console.WriteLine("Graceful shutdown initiated...");
});

await app.RunAsync();

    X. Dev/Prod Parity üîÅ

    factor 10 environments parity

    Core Principle

    "Keep development, staging, and production as similar as possible."

    Historically, there have been substantial gaps between development and production. These gaps manifest in three areas. The twelve-factor app is designed for continuous deployment by keeping the gap between development and production small.

    The Three Gaps

    GapTraditional AppTwelve-Factor App
    ‚è±Ô∏è Time gapCode takes weeks or months to reach productionDeploy in hours or even minutes
    üë• Personnel gapDevelopers write code, ops engineers deploy itDevelopers who wrote code are closely involved in deploying and watching behavior in production
    üîß Tools gapDevelopers use Nginx, SQLite, OS X; production uses Apache, MySQL, LinuxKeep development and production as similar as possible

    Backing Services: The Tools Gap in Practice

    Developers sometimes use a lightweight backing service locally while a more robust one runs in production (e.g., SQLite locally and PostgreSQL in production). The twelve-factor developer resists this urge, even when adapters theoretically abstract away differences.

    ‚ö†Ô∏è Why This Matters: Differences between backing services mean tiny incompatibilities crop up ‚Äî code that worked locally or in staging fails in production. This creates friction that disincentivizes continuous deployment. The cost over the lifetime of an application is extremely high.

    XI. Logs üìã

    factor 11 observability streams

    Core Principle

    "Treat logs as event streams."

    Logs are the stream of aggregated, time-ordered events collected from the output streams of all running processes and backing services. Logs in their raw form are typically a text format with one event per line (though backtraces from exceptions may span multiple lines). Logs have no fixed beginning or end, but flow continuously as long as the app is operating.

    The 12-Factor Approach

    A twelve-factor app never concerns itself with routing or storage of its output stream. It should not attempt to write to or manage logfiles. Instead:

    • Each running process writes its event stream, unbuffered, to stdout
    • During local development, the developer views this stream in the terminal foreground to observe the app's behavior
    • In staging or production deploys, the execution environment captures each process' stream, collates all streams together, and routes to one or more final destinations for viewing and long-term archival
    üîç Key Insight: Log routing destinations are not visible to or configurable by the app ‚Äî they are completely managed by the execution environment. The app writes to stdout and is done.

    • Fluentd ‚Äî open-source data collector for unified logging

    What You Can Do Once Logs Are Streams

    • Route the event stream to a file, or watch via realtime tail in a terminal
    • Send to a log indexing and analysis system such as Splunk
    • Send to a general-purpose data warehousing system such as Hadoop/Hive
    • Find specific events in the past
    • Large-scale graphing of trends (e.g., requests per minute)
    • Active alerting based on user-defined heuristics (e.g., alert when quantity of errors per minute exceeds a certain threshold)

    XII. Admin Processes üõ†Ô∏è

    factor 12 tasks maintenance

    Core Principle

    "Run admin/management tasks as one-off processes."

    The process formation is the array of processes used to do the app's regular business (such as handling web requests). Separately, developers will often wish to do one-off administrative or maintenance tasks, such as:

    • Running database migrations (e.g., manage.py migrate in Django, rake db:migrate in Rails)
    • Running a console (REPL shell) to run arbitrary code or inspect the app's models against the live database (e.g., python, irb for Ruby, rails console)
    • Running one-time scripts committed into the app's repo (e.g., php scripts/fix_bad_records.php)

    Same Environment, Same Release

    One-off admin processes should be run in an identical environment as the regular long-running processes of the app. They run against a release, using the same codebase and config as any process run against that release.

    üîë Critical Rule: Admin code must ship with application code to avoid synchronization issues. Admin scripts are not separate, ad-hoc scripts maintained outside the application.

    Same Dependency Isolation

    The same dependency isolation techniques should be used on all process types:

    • If the Ruby web process uses bundle exec thin start, then a database migration should use bundle exec rake db:migrate
    • A Python program using Virtualenv should use the vendored bin/python for running both the Tornado webserver and any manage.py admin processes

    Task TypeExampleTool
    DB MigrationApply schema changesdotnet ef database update, flyway migrate
    REPL ConsoleInspect live databaserails console, python, irb
    Data SeedingLoad reference dataCustom script in same codebase
    Cleanup JobPurge old recordsCLI command run as Kubernetes Job

    How to Run Admin Processes

    • Local deploy: Invoke via a direct shell command inside the app's checkout directory
    • Production deploy: Use ssh or other remote command execution mechanism provided by the deploy's execution environment
    • Kubernetes: Use kubectl exec or Kubernetes Jobs

    üö´ Anti-Patterns

    violations pitfalls what not to do

    Common Violations Per Factor

    These are the most frequently observed real-world violations of the 12-factor methodology, and why they cause problems at scale.

    Factor‚ùå Common Violationüí• Why It Hurts‚úÖ Correct Approach
    I. Codebase Multiple apps in one repo (monorepo deployed as one unit), or one app split across multiple repos Coupled deployments; impossible to deploy apps independently One codebase per deployable app; share code via libraries as packages
    II. Dependencies Relying on system-wide tools (curl, ImageMagick, python) being pre-installed Works on developer's machine, fails in CI/CD or production Vendor or declare all tools; use Docker to capture the full execution environment
    III. Config Hardcoding credentials, API keys, or URLs in source code, or using config/dev.yml / config/prod.yml files committed to the repo Security breach risk; config-per-environment files explode with new environments Env vars only (or a secrets manager like Azure Key Vault / AWS Secrets Manager)
    IV. Backing Services Hardcoding database connection strings or treating a local RabbitMQ as fundamentally different from a cloud-hosted one Cannot swap services without code changes; tight coupling to infrastructure All resources accessed via URLs/credentials in config ‚Äî swappable with zero code changes
    V. Build/Release/Run SSH-ing into a production server and patching code directly (cowboy deploy); or building inside a running container No audit trail; impossible to roll back; state drift between instances Immutable releases via CI/CD pipeline; rollback by redeploying a previous release ID
    VI. Processes Storing session state in process memory (sticky sessions); caching user data on local disk Breaks horizontal scaling; a restart loses all in-memory state Store session in Redis/Cosmos DB; use distributed cache; all process instances are interchangeable
    VII. Port Binding App requires Tomcat/IIS/Apache to be separately installed in the environment Environment coupling; harder to containerise; can't run locally without full app-server setup Bundle the web server as a library dependency (Kestrel, Jetty, Gunicorn)
    VIII. Concurrency Handling all traffic with threads inside a single process; using PID files or daemonising Single process can't scale horizontally; crashes take down all capacity Use the process model: scale by adding more processes, managed by systemd/Kubernetes
    IX. Disposability Long startup times (30+ seconds); not handling SIGTERM; leaving jobs locked in a queue on crash Slow scaling; data loss on crash; stuck queue messages Fast startup; SIGTERM handlers; idempotent/reentrant jobs; robust queuing (Beanstalkd, Service Bus)
    X. Dev/Prod Parity SQLite in dev, PostgreSQL in prod; local in-memory cache vs. Memcached in prod Bugs only appear in production; false confidence from green local tests Docker Compose / Testcontainers to run the same backing services locally as production
    XI. Logs Writing logs to /var/log/app.log; rotating log files from within the app; suppressing logs in production Logs lost on container restart; no central aggregation; blind in production Write unbuffered to stdout; let the platform (Fluentd, Azure Monitor) handle routing and storage
    XII. Admin Processes Maintaining a separate maintenance scripts repo; running migrations from a dev laptop against production; ad-hoc SQL run directly on prod DB Sync issues between app and scripts; untested in CI; no audit trail Admin scripts in the same codebase; run as one-off processes using the same Docker image & env vars

    üîÑ Beyond 12-Factor

    evolution cloud-native 15 factors

    The Evolution

    The original 12-factor methodology was published by Adam Wiggins in 2011, based on Heroku's experience with early SaaS applications. In 2016, Kevin Hoffman published "Beyond the 12-Factor App" (O'Reilly), updating the methodology for the cloud-native era of containers, microservices, and cloud-managed backing services.

    üìÖ Context Shift (2011 ‚Üí 2016+): The original 12-factor predate Docker, Kubernetes, cloud-managed databases, and serverless. The "Beyond" additions reflect how the landscape evolved.

    The Three Additional Factors

    #FactorCore Idea
    XIII API First Design the API contract before implementation (API-first / contract-first). In a cloud-native system, services communicate over APIs. The API is the product. Use OpenAPI/Swagger, AsyncAPI, or gRPC contracts defined up front, enabling teams to build consumers and providers in parallel.
    XIV Telemetry Every app should emit three signals: logs (events), metrics (counters/gauges/histograms), and traces (distributed spans). Factor XI covers logs ‚Äî Telemetry extends this to the full observability triad: logs + metrics + traces (OpenTelemetry, Application Insights, Prometheus, Jaeger).
    XV Authentication & Authorization Security must be built in from the start, not bolted on. Every app in a cloud-native system must implement proper AuthN/AuthZ. In practice: use standards like OAuth 2.0 / OIDC, delegate to an identity provider (Azure Entra ID / Auth0), never roll your own auth, and ensure every inter-service call is authenticated.
