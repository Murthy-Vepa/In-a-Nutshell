    What is ArchiMate?

    ArchiMate¬Æ is an open and independent enterprise architecture modeling language that provides a common visual notation and grammar for describing, analyzing, and communicating architecture across business domains, information systems, and technology infrastructure.

    üéØ Key Definition

    ArchiMate is a visual language with a common lexicon of shapes and connectors that enables architects to describe, analyze, and visualize relationships among business domains, information systems, and technology infrastructure within an enterprise.

    Purpose & Scope

    ArchiMate was created by The Open Group to address the need for a standardized way to describe enterprise architectures. It serves as:

    • A Common Language - Enables clear communication among stakeholders including business leaders, architects, developers, and operations teams
    • A Modeling Standard - Provides consistent notation and semantics for representing architectural concepts
    • An Analysis Tool - Supports impact analysis, dependency tracking, and architecture assessment
    • A Documentation Framework - Facilitates comprehensive architecture documentation and knowledge management
    ‚úÖ What ArchiMate IS
    • Enterprise architecture modeling language
    • Visual communication tool
    • Standardized notation system
    • Framework for architecture analysis
    • Tool-independent specification
    ‚ö†Ô∏è What ArchiMate is NOT
    • Not a software tool (it's a language/standard)
    • Not a detailed design notation (like UML)
    • Not a process modeling language (like BPMN)
    • Not an architecture framework (like TOGAF)
    • Not limited to IT architecture only

    The ArchiMate Standard

    ArchiMate is maintained by The Open Group, a global consortium that develops open, vendor-neutral technology standards. The standard defines:

    • Core Concepts - Elements (components) and relationships (connections)
    • Layers - Business, Application, Technology, Strategy, and Motivation
    • Aspects - Active structure, behavior, and passive structure
    • Viewpoints - Predefined perspectives for different stakeholders
    • Notation - Visual symbols and conventions for modeling

    Common Anti-Patterns (What to Avoid)

    ‚ùå Using ArchiMate for Everything
    Don't try to model every detail in ArchiMate. Keep it at the architectural level and use specialized tools for detailed design, process flows, or data modeling.
    ‚ùå Over-Complicating Simple Scenarios
    For quick discussions or one-off explanations, a whiteboard sketch might be more effective than creating a formal ArchiMate model.
    ‚ùå Modeling Without Purpose
    Always start with the question: "What decision or communication does this model support?" Don't model just for the sake of having models.
    ‚ùå Ignoring Stakeholder Needs
    ArchiMate models should be tailored to stakeholder concerns. A single, complex model trying to serve everyone usually serves no one well.
    üí° Golden Rule

    Use ArchiMate when you need to communicate architecture across organizational boundaries and maintain that documentation over time. For everything else, evaluate if simpler alternatives might be more appropriate.

    2. Behavior Aspect

    Behavior elements are the "verbs" in your architecture story - the actions, processes, and services.

    üìã

    Process

    Ordered sequence of activities with a start and end. Has temporal ordering.

    Example: Order Fulfillment Process

    ‚öôÔ∏è

    Function

    Collection of behavior without explicit ordering. Continuous or grouped.

    Example: Inventory Management

    üéÅ

    Service

    External behavior exposed to environment. Black-box view.

    Example: Order Submission Service

    Key Distinction: Internal vs External View
    • Processes & Functions = Internal view (how things work inside)
    • Services = External view (what's offered to the outside)
    • Services are realized by processes and functions
    ‚ö†Ô∏è Common Confusion: Process vs Function

    Process implies temporal ordering (step 1, then step 2, then step 3). Function is a collection of related behaviors without mandated sequence. Use Process when order matters!

    Purpose of Structural Relationships

    Structural relationships model the static structure of the architecture by expressing how elements are composed, grouped, assigned, or realized.

    Key Principle: Structural relationships define "is part of" and "implements" connections, forming the architectural hierarchy and structure.

    Composition Relationship

    ‚óÜ‚Äî‚Äî Composition

    Definition: Indicates that an element consists of one or more other concepts. The composed element is part of the composition and cannot exist without it.

    Notation: Solid line with filled diamond at the "whole" end

    Strength: Strong - Lifecycle dependency (part cannot exist without whole)

    Read as:
    "consists of"
    "is composed of"
    "contains"
    Common Examples:
    • Business Process composed of Sub-Processes
    • Application Component composed of Sub-Components
    • Node composed of Devices
    • Business Service composed of Sub-Services
    • Business Function composed of Business Functions
    • Work Package composed of Sub-Work Packages
    üí° When to use: Use composition when the parts have NO independent meaning without the whole. If you delete the parent, the children cease to exist in the model.

    Aggregation Relationship

    ‚óá‚Äî‚Äî Aggregation

    Definition: Indicates that an element combines or groups one or more other concepts. The aggregated elements can exist independently.

    Notation: Solid line with hollow diamond at the "whole" end

    Strength: Moderate - No lifecycle dependency (parts can exist independently)

    Read as:
    "aggregates"
    "groups"
    "combines"

    • Location aggregates Business Actors
    • Grouping aggregates multiple Elements
    • Product aggregates Services
    • Plateau aggregates Elements (snapshot)
    • Course of Action aggregates Capabilities
    • Business Collaboration aggregates Business Roles
    üí° When to use: Use aggregation when elements are grouped for organizational purposes but can exist independently. If you delete the parent, children continue to exist.

    Understanding Derived Relationships

    A derived relationship is an indirect relationship inferred from a chain of direct relationships. ArchiMate defines specific rules about which relationship chains create valid derived relationships.

    Key Principle: Derived relationships allow you to simplify complex diagrams by showing indirect connections without cluttering the view with intermediate elements.

    Why Derived Relationships Matter

    ‚úÖ Benefits
    • Simplified views: Hide intermediate layers for stakeholder-specific views
    • Impact analysis: Trace dependencies across multiple layers
    • Flexibility: Show same architecture at different abstraction levels
    • Validation: Tools can compute and validate derived relationships
    ‚ö†Ô∏è Considerations
    • Tool support: Not all tools compute derived relationships automatically
    • Complexity: Long chains can be hard to understand
    • Not all chains valid: ArchiMate has specific derivation rules
    • Explicit vs derived: Must distinguish in models

    Common Derived Relationship Patterns

    Pattern 1: Cross-Layer Service Derivation

    Scenario: Showing how technology services ultimately support business processes

    Direct Relationships:
    1. Technology Service realizes Technology Function
    2. Technology Function serves Application Component
    3. Application Component realizes Application Service
    4. Application Service serves Business Process
    Derived Relationship:

    Technology Service (indirectly) serves Business Process

    This allows a business-focused view showing technology support without showing all intermediate layers.

    Pattern 2: Assignment Chain Derivation

    Scenario: Showing which actors ultimately perform which processes

    Direct Relationships:
    1. Business Actor assigned to Business Role
    2. Business Role assigned to Business Function
    3. Business Function assigned to Business Process
    Derived Relationship:

    Business Actor (indirectly) assigned to Business Process

    Useful for showing responsibilities without intermediate role and function layers.

    Pattern 3: Realization Chain Derivation

    Scenario: Showing how low-level implementations support high-level services

    Direct Relationships:
    1. Application Function realizes Application Service
    2. Application Service realizes Business Service
    Derived Relationship:

    Application Function (indirectly) realizes Business Service

    Shows direct implementation path from function to external business service.

    Derivation Rules Summary

    Relationship 1 + Relationship 2 = Derived Relationship
    Realization + Serving = Serving (derived)
    Serving + Serving = Serving (derived)
    Assignment + Assignment = Assignment (derived)
    Composition + Composition = Composition (derived)
    Aggregation + Aggregation = Aggregation (derived)
    Access (Read) + Realization = Access (derived)
    Influence + Influence = Influence (derived)

    Not all relationship chains create valid derived relationships. For example:

    • Composition + Serving ‚â† Serving - Cannot derive serving from composition
    • Triggering + Realization ‚â† Triggering - Dynamic relationships don't propagate through realization
    • Flow + Assignment ‚â† Flow - Flow doesn't derive through assignment

    Working with Derived Relationships

    ‚úÖ Do
    • Use derived relationships for simplified stakeholder views
    • Leverage for impact analysis and traceability
    • Distinguish visually from direct relationships (different line style or color)
    • Let tools compute derived relationships automatically when possible
    • Document derivation chains for complex cases
    ‚ùå Don't
    • Don't show both direct and derived relationships in same view (cluttered)
    • Don't create invalid derivation chains
    • Don't rely on derived relationships for detailed design (use direct relationships)
    • Don't forget to maintain underlying direct relationships
    • Don't assume all tools support automatic derivation
    üîß Tool Support

    ‚úÖ Best Practices for Derived Relationships

    • Impact analysis: Use derived relationships to trace dependencies across layers (e.g., "What business processes depend on this server?")
    • Visual distinction: Always distinguish derived from direct relationships (different color, line style, or annotation)
    • Documentation: When showing derived relationships, document the underlying direct relationship chain
    • Validation: Use tools to validate that derived relationships follow ArchiMate rules

    Understanding Viewpoints and Views

    ArchiMate distinguishes between viewpoints (templates/specifications) and views (actual model visualizations). This distinction is crucial for effective architecture communication.

    üìã Viewpoint

    Definition: A specification of the perspective from which to look at the architecture, defining what to show and what to omit.

    Think of it as: A template or recipe for creating views

    Specifies:

    • Which element types to include
    • Which relationship types to show
    • Target stakeholders
    • Concerns addressed
    üìä View

    Definition: A representation of a system from the perspective of a specific viewpoint, containing actual architecture model elements.

    Think of it as: An instance or diagram based on a viewpoint

    Contains:

    • Actual elements from your model
    • Actual relationships
    • Specific to your organization
    • Concrete visualization
    Analogy: A viewpoint is like a recipe (tells you what ingredients to use), while a view is the actual meal you prepare following that recipe with your specific ingredients.

    Why Viewpoints Matter

    üéØ The Problem Viewpoints Solve

    A complete enterprise architecture model can contain hundreds or thousands of elements. Showing everything to everyone is:

    ‚ùå Overwhelming
    Too much information to process
    ‚ùå Irrelevant
    Not all stakeholders need all details
    ‚ùå Confusing
    Mixed concerns and perspectives

    Solution: Viewpoints allow you to create targeted views that show exactly what specific stakeholders need to see, addressing their specific concerns.

    Common Modeling Pitfalls

    Symptom: Models with excessive detail that obscure the important information

    Causes:

    • Modeling without clear purpose ("just in case we need it")
    • Trying to document everything in one model
    • Fear of leaving something out

    Solution:

    • Always define purpose before modeling
    • Create multiple focused models instead of one complex model
    • Use abstraction‚Äîhide unnecessary details
    • Apply the "minimal effective model" principle

    Symptom: Same concept called different things; confusion about whether elements are the same

    Examples:

    • "Customer", "Client", "Customer Entity" referring to same thing
    • "Order Management System" vs "OMS" vs "Order System"
    • Inconsistent capitalization or abbreviations

    Solution:

    • Establish naming conventions early
    • Create and maintain a glossary
    • Use element library for reuse
    • Review models for naming consistency
    • Use full names rather than abbreviations when clarity matters

    Symptom: Models mixing high-level strategy with low-level implementation details

    Examples:

    • Strategic capability model showing database tables
    • Executive dashboard with technical infrastructure details
    • High-level process model with specific field names

    Solution:

    • Match abstraction level to stakeholder needs
    • Create separate models for different levels
    • Use viewpoints to control what's shown
    • Provide drill-down paths from high to low level

    Symptom: Relationships that don't match ArchiMate metamodel rules

    Common Mistakes:

    • Using composition when aggregation is correct (or vice versa)
    • Using association for everything ("it's related somehow")
    • Incorrect direction of serving relationships
    • Realizations going the wrong way

    Solution:

    • Learn the precise meaning of each relationship type
    • Use metamodel as reference for valid relationships
    • When uncertain, check ArchiMate specification
    • Use tool validation features if available

    Symptom: Models never finished because they're "not perfect yet"

    Causes:

    • Waiting to capture every detail
    • Endless refinement without clear completion criteria
    • Fear of producing "wrong" models

    Solution:

    • Remember: "All models are wrong, but some are useful"
    • Define "done" criteria upfront
    • Release models iteratively‚Äî"good enough for now"
    • Plan for updates rather than initial perfection
    • Focus on fitness for purpose, not absolute perfection

    Symptom: Models become outdated and lose credibility

    Impact:

    • Stakeholders stop trusting models
    • Models provide misleading information
    • Investment in modeling is wasted

    Solution:

    • Establish model governance and ownership
    • Schedule regular review cycles
    • Update models when architecture changes
    • Use version control for model history
    • Retire obsolete models explicitly

    üîÑ BPMN Integration

    bpmn process modeling

    BPMN (Business Process Model and Notation) is the standard for detailed business process modeling. While ArchiMate includes business processes, BPMN provides much richer process semantics. Understanding when to use each‚Äîand how they complement each other‚Äîenables effective process architecture.

    ArchiMate vs BPMN: Different Purposes

    üî∑ ArchiMate Business Process

    Purpose: Architecture and context

    • High-level: What processes exist and how they relate
    • Context-rich: Who performs processes, which apps support them
    • Cross-layer: Link processes to applications and technology
    • Stable: Relatively unchanging architecture view
    • Strategic: Enterprise-wide process landscape

    Use for: Process architecture, organizational context, EA documentation

    üîÑ BPMN Process

    Purpose: Detailed process execution

    • Detailed: Step-by-step activities, gateways, events
    • Executable: Can be executed by process engines (workflow)
    • Process-focused: Internal process logic and flow control
    • Precise: Unambiguous execution semantics
    • Operational: Day-to-day process operations

    Use for: Process design, automation, operational procedures, BPM systems

    Element Mapping: ArchiMate ‚Üî BPMN

    ArchiMate Element BPMN Element Mapping Notes
    Business Process Process (with multiple activities) ArchiMate process = entire BPMN process diagram; represents process at architectural level
    Business Function Sub-Process or Activity BPMN activities can map to ArchiMate functions when representing reusable behaviors
    Business Actor / Role Pool / Lane BPMN pools represent organizations; lanes represent roles within organization
    Business Service Message Flow (received) Service calls in BPMN map to service consumption in ArchiMate
    Business Event Start/End/Intermediate Events Both represent occurrences that trigger or result from processes
    Business Object Data Object Both represent information used by processes
    Triggering Relationship Sequence Flow Both show temporal dependencies, but BPMN has richer flow semantics
    Application Service Service Task BPMN service tasks invoke application services modeled in ArchiMate

    Integration Patterns

    Approach: Use ArchiMate to show the process landscape and organizational context, then drill down to BPMN for process execution details.

    ArchiMate Model Shows:

    • All processes in the organization
    • Which roles perform which processes
    • Which applications support processes
    • How processes relate to business services
    • Strategic importance of processes

    BPMN Model Shows:

    • Step-by-step activities within each process
    • Decision points and gateways
    • Exceptions and error handling
    • Timing and performance requirements
    • Precise execution logic

    Best for: Enterprise process architecture with selective detailed process design

    Approach: Start with detailed BPMN process models, then import/transform them into ArchiMate business layer for enterprise architecture.

    Transformation Rules:

    • BPMN Process ‚Üí ArchiMate Business Process
    • BPMN Pool/Lane ‚Üí ArchiMate Business Actor/Role
    • BPMN Data Object ‚Üí ArchiMate Business Object
    • BPMN Service Task ‚Üí ArchiMate Application Service (accessed by process)
    • Flatten BPMN detail‚Äîfocus on high-level process structure

    Best for: Organizations with existing BPMN models wanting enterprise architecture view

    Approach: Start with ArchiMate business architecture, then generate BPMN process templates for detailed design.

    Generation Rules:

    • ArchiMate Business Process ‚Üí BPMN Process (skeleton)
    • ArchiMate Triggering chains ‚Üí BPMN Sequence flows
    • ArchiMate Business Actor ‚Üí BPMN Pool
    • ArchiMate Application Service usage ‚Üí BPMN Service Task
    • Process designers then add BPMN detail (gateways, events, etc.)

    Best for: Top-down process design starting from enterprise architecture

    Approach: Maintain separate ArchiMate and BPMN models with explicit traceability links between them.

    ArchiMate Model:

    • Contains Business Process elements
    • Each process has property/metadata pointing to BPMN model
    • Shows organizational and application context
    • Maintained by enterprise architects

    BPMN Model:

    • Contains detailed process logic
    • References corresponding ArchiMate process
    • Shows execution details
    • Maintained by process analysts/designers

    Best for: Mature organizations with both EA and BPM practices; tool support for linking

    üìê UML Relationship

    uml design software

    UML (Unified Modeling Language) is the standard for object-oriented software design. While ArchiMate models enterprise architecture, UML focuses on detailed software design. Understanding their relationship enables seamless transition from architecture to implementation.

    ArchiMate vs UML: Complementary Scopes

    üî∑ ArchiMate

    Focus: Enterprise Architecture

    • Scope: Business, application, technology‚Äîenterprise-wide
    • Abstraction: Architectural level, conceptual to logical
    • Audience: Enterprise architects, business stakeholders
    • Purpose: Strategic alignment, system integration, impact analysis
    • Granularity: Systems, services, components
    • Lifecycle: Architecture evolution over years
    üìê UML

    Focus: Software Design & Implementation

    • Scope: Software systems‚Äîinternal structure and behavior
    • Abstraction: Logical to physical design, code level
    • Audience: Software architects, developers, designers
    • Purpose: Detailed design, code generation, documentation
    • Granularity: Classes, methods, objects, interactions
    • Lifecycle: Software releases, iterations (weeks/months)
    üí° The Relationship

    ArchiMate stops where UML begins. ArchiMate models applications as components and services at the architecture level. UML then designs the internal structure of those applications with classes, interfaces, and interactions.

    Element Mapping: ArchiMate ‚Üî UML

    ArchiMate Element UML Diagram / Element Mapping Explanation
    Application Component Component, Package ArchiMate component = UML component or package; high-level software unit
    Application Service Interface (provided) Services in ArchiMate = interfaces in UML; external behavior
    Application Interface Interface, Port Both represent access points to component functionality
    Application Function Method, Operation Internal behaviors; UML operations implement ArchiMate functions
    Data Object Class (entity), Data Type ArchiMate data objects become UML entity classes or data structures
    Application Interaction Sequence Diagram, Communication Diagram ArchiMate shows that interaction occurs; UML details the message exchanges
    Application Process Activity Diagram, State Machine ArchiMate process at app level; UML shows detailed behavior
    Artifact Artifact, Component (deployment) Both represent physical deployment units (JARs, DLLs, executables)
    Technology Node Node (Deployment Diagram) Both represent hardware or execution environments

    UML Diagram Types & ArchiMate Views

    UML Class Diagram Shows:

    • Classes with attributes and methods
    • Inheritance hierarchies
    • Associations and multiplicities
    • Interfaces and implementations
    • Detailed OO design

    ArchiMate Application View Shows:

    • Application components (packages/modules)
    • Services provided/required
    • Data objects (high-level entities)
    • Component dependencies
    • Architectural structure

    Usage: Start with ArchiMate to define component boundaries, then use UML class diagrams to design internal structure of each component.

    UML Component Diagram Shows:

    • Software components (JARs, libraries)
    • Provided/required interfaces
    • Component wiring and dependencies
    • Technical deployment structure

    ArchiMate Cooperation View Shows:

    • Application components and services
    • Service dependencies and flows
    • Integration patterns
    • Architectural cooperation

    Overlap: UML component diagrams are very similar to ArchiMate application views. UML is often more technical; ArchiMate more business-oriented.

    UML Sequence Diagram Shows:

    • Time-ordered message exchanges
    • Object lifelines and activations
    • Synchronous/asynchronous calls
    • Return values and parameters
    • Detailed interaction logic

    ArchiMate Flow View Shows:

    • High-level service calls
    • Data flows between components
    • Triggering relationships
    • Application cooperation patterns
    • Architectural interaction

    Usage: ArchiMate shows that services interact; UML sequence diagrams detail the exact message exchanges.

    UML Deployment Diagram Shows:

    • Nodes (hardware devices, servers)
    • Artifacts (deployed files)
    • Manifestation (which artifacts deploy where)
    • Communication paths

    ArchiMate Technology View Shows:

    • Technology nodes and devices
    • Artifacts (deployable units)
    • Assignment of artifacts to nodes
    • Network paths and communication

    Similarity: UML deployment diagrams and ArchiMate technology views are very similar‚Äîboth model physical deployment.

    Integration Approaches

    Approach 1: Top-Down (Architecture ‚Üí Design)
    1. Start with ArchiMate: Define application components and services at architecture level
    2. Establish boundaries: Determine component interfaces and responsibilities
    3. Generate UML skeletons: Create UML packages and interfaces from ArchiMate components
    4. Detailed UML design: Designers flesh out classes, methods, interactions
    5. Maintain traceability: Link UML artifacts back to ArchiMate elements

    Best for: Greenfield projects, architecture-first organizations

    Approach 2: Bottom-Up (Design ‚Üí Architecture)
    1. Start with UML: Developers create detailed class and component designs
    2. Group into components: Identify cohesive packages as architectural components
    3. Extract interfaces: Promote key interfaces to architectural services
    4. Create ArchiMate view: Model high-level architecture from UML designs
    5. Refine and rationalize: Clean up architecture based on implementation reality

    Best for: Existing systems, reverse engineering, design-first cultures

    ArchiMate Application Architecture

    üìö Glossary

    terms definitions vocabulary

    A comprehensive glossary of ArchiMate terminology and related enterprise architecture concepts. Organized alphabetically for easy reference.

    Note: Definitions are based on ArchiMate 3.2 specification and common EA practices. For authoritative definitions, refer to The Open Group ArchiMate¬Æ 3.2 Specification.

    A

    Access Relationship
    A dependency relationship that models the ability of behavior and active structure elements to observe or act upon passive structure elements (data objects, artifacts, business objects).
    Active Structure Element
    An element that is capable of performing behavior. Examples: Business Actor, Application Component, Node.
    Aggregation Relationship
    A structural relationship that indicates an element groups a number of other elements, without implying strong ownership (compare to composition).
    Application Component
    An encapsulation of application functionality aligned to implementation structure. A modular, deployable, and replaceable part of a software system.
    Application Layer
    The layer that supports the business layer with application services realized by software applications and data objects.
    ArchiMate
    An open and independent enterprise architecture modeling language from The Open Group that supports the description, analysis, and visualization of architecture across business domains.
    Artifact
    A physical piece of data that is used or produced in a software development process, or by deployment and operation of an IT system. Examples: executable file, library, database table.
    Assignment Relationship
    A structural relationship that allocates responsibility for performing behavior to an active structure element, or for storing data to a passive structure element.
    Association Relationship
    A dependency relationship that models an unspecified relationship between elements. Should be used sparingly when no other relationship type fits.

    B

    Behavior Element
    An element that represents a unit of activity performed by one or more active structure elements. Examples: Process, Function, Service, Event.
    Business Actor
    An organizational entity capable of performing behavior. Examples: person, department, company, organizational unit.
    Business Layer
    The architectural layer that models business services offered to customers, realized by business processes performed by business actors and roles.
    Business Object
    A passive element representing a concept used within a business domain. Examples: customer, order, invoice, policy.
    Business Process
    A sequence of business behaviors that produces a specified outcome. A unit of collective behavior performed by business roles or actors.
    Business Service
    A service that fulfills a business need for a customer (internal or external). Explicitly exposes business functionality to the environment.

    C

    Capability
    An ability that an active structure element (organization, person, system) possesses. Represents a high-level business ability independent of how it's realized.
    Collaboration
    An aggregate of two or more active structure elements that work together to perform collective behavior. Example: project team, working group.
    Composition Relationship
    A structural relationship that represents a whole is composed of one or more parts. Implies strong ownership‚Äîif the whole is deleted, parts are also deleted.
    Constraint
    A restriction on the way in which a system or enterprise may be realized. A factor that prevents or obstructs the realization of goals.
    Course of Action
    An approach or plan for configuring capabilities and resources to achieve a goal. Similar to a strategy or initiative.

    D

    Data Object
    A passive structure element in the application layer that represents data structures. Examples: database table, XML schema, data entity.
    Deliverable
    A precisely defined result of a work package. A tangible output from a project or implementation effort.
    Dependency Relationship
    A category of relationships (serving, access, influence, association) that describe how elements depend on or use each other.
    Derived Relationship
    An indirect relationship that can be inferred from a chain of other relationships. ArchiMate defines rules for deriving valid relationships.
    Device
    A physical IT resource (hardware) upon which system software and artifacts may be deployed for execution. Examples: server, workstation, mobile device.
    Driver
    Something that creates, motivates, or fuels change in an organization. External or internal factor that influences goals. Examples: regulation, market trend, technology shift.
    Dynamic Relationship
    A category of relationships (triggering, flow) that describe temporal dependencies and information flows between behavior elements.

    E-F

    Event
    A behavior element representing a state change or occurrence that triggers or results from behavior. Examples: Order Received, Payment Completed.
    Flow Relationship
    A dynamic relationship that represents transfer of information, goods, or value between behavior elements.
    Function
    A collection of behavior based on a chosen set of criteria (typically required resources and competencies). More static than a process.

    G

    Gap
    A statement of difference between two plateaus. Used in migration planning to identify what needs to change between current and target states.
    Goal
    A high-level statement of intent, direction, or desired end state for an organization. Represents "what" the organization wants to achieve.
    Grouping
    An aggregation of elements of the same or different types. Used purely for organizational purposes without implying semantic relationships.

    I

    Implementation Layer
    A layer supporting implementation and migration of architectures through work packages, deliverables, and plateaus.
    Influence Relationship
    A dependency relationship that describes how motivation elements (drivers, goals, assessments) affect or modify other elements.
    Interface
    A point of access where application or technology services are made available to users or other components.
    Interaction
    A unit of collective behavior performed by two or more structure elements. Describes cooperation or communication between elements.

    J-L

    Junction
    An element used to connect relationships of the same type to combine or split them using AND (all required) or OR (one of) semantics.
    Layer
    A grouping of elements at the same level of abstraction. ArchiMate defines core layers (Business, Application, Technology) plus supporting layers (Strategy, Motivation, Implementation, Physical).
    Location
    A place where behavior is performed or structure elements are positioned. Can be physical (building, city) or virtual (network zone).

    M-N

    Material
    Tangible physical matter or energy in the physical layer. Examples: raw materials, water, electricity.
    Meaning
    The knowledge or expertise present in a business object or its representation, given a particular context. Interprets the value of information.
    Metamodel
    A model of ArchiMate itself; defines the concepts (elements and relationships) and rules for creating valid ArchiMate models.
    Motivation Layer
    A layer that models the reasons and motivations that guide the design or change of an enterprise architecture.
    Network
    A communication medium between two or more devices or nodes. Examples: LAN, WAN, internet connection.
    Node
    A computational or physical resource upon which artifacts may be deployed for execution. Abstract concept; specialized by Device and System Software.

    O-P

    Outcome
    An end result that has been achieved. Often used to measure success of goals. More concrete than a goal.
    Passive Structure Element
    An element on which behavior is performed. Cannot perform behavior itself. Examples: Business Object, Data Object, Artifact, Material.
    Path
    A link between two or more nodes through which information or materials can be exchanged. Lower-level than network.
    Physical Layer
    A layer representing physical elements like equipment, facilities, distribution networks, and materials.
    Plateau
    A relatively stable, incremental state of the architecture representing a defined point in an evolution timeline. Used in migration planning.
    Principle
    A normative property or guideline that shapes the intentions or behavior of an organization. Example: "Buy before build".
    Product
    A coherent collection of services and/or passive structure elements, accompanied by a contract/set of agreements, offered as a whole to customers.

    R

    Realization Relationship
    A structural relationship indicating that an entity plays a critical role in the creation, achievement, sustenance, or operation of a more abstract entity.
    Relationship
    A connection between two or more elements. ArchiMate defines specific relationship types with precise semantics.
    Representation
    A perceptible form of information carried by a business object. Example: paper document, electronic file, verbal communication.
    Requirement
    A statement of need defining a property that applies to a specific system or element. More specific than a goal.
    Resource
    An asset owned or controlled by an individual or organization. Examples: people, budget, equipment, information.
    Role
    The responsibility for performing specific behavior to which an actor can be assigned. Separates "what" from "who".

    S

    Service
    A unit of functionality that provides value to its environment. Explicitly defines externally visible behavior. Can be business, application, or technology service.
    Serving Relationship
    A dependency relationship that represents that an element provides its functionality to another element. The primary dependency relationship in ArchiMate.
    Specialization Relationship
    A relationship indicating that an element is a particular kind of another element (subtype). Similar to inheritance in OO modeling.
    Stakeholder
    The role of an individual, team, or organization that represents their interests in the effects of the architecture.
    Strategy Layer
    A layer modeling strategic direction through elements like capability, resource, value stream, and course of action.
    System Software
    Software that provides or contributes to an environment for storing, executing, and using software or data. Examples: OS, database management system, middleware.

    T

    Technology Layer
    The layer offering infrastructure services (processing, storage, communication) needed to run applications.
    TOGAF
    The Open Group Architecture Framework‚Äîa framework for enterprise architecture providing a method (ADM) for developing and managing architectures. ArchiMate complements TOGAF.
    Triggering Relationship
    A dynamic temporal relationship indicating that one behavior element initiates another behavior element. Represents causal dependencies.

    V

    Value
    The relative worth, utility, or importance of a service or product. Represents what stakeholders gain.
    Value Stream
    A sequence of activities that create an overall result for a customer, stakeholder, or end user. Focuses on value creation.
    View
    A representation of a system from the perspective of a related set of concerns. A subset of the model tailored to specific stakeholders.
    Viewpoint
    A specification of the content and structure of a view. Defines which elements and relationships can appear in a view and for what stakeholders.

    W

    Work Package
    A series of actions designed to accomplish a unique goal within a specified time. Used in implementation and migration planning.
