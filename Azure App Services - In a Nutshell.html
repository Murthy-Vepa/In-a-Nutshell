    üìñ Overview

    intro overview basics

    Azure App Service is an HTTP-based service for hosting web applications, REST APIs, and mobile back ends. You can develop in your favorite language, be it .NET, .NET Core, Java, Node.js, PHP, or Python. Applications run and scale with ease on both Windows and Linux-based environments.

    üéØ Core Purpose
    App Service not only adds the power of Microsoft Azure to your application, such as security, load balancing, autoscaling, and automated management, but also enables DevOps capabilities like continuous deployment from Azure DevOps, GitHub, Docker Hub, and other sources, package management, staging environments, custom domain, and TLS/SSL certificates.

    Why App Service?

    üöÄ Multiple Languages and Frameworks

    App Service has first-class support for ASP.NET, ASP.NET Core, Java, Ruby, Node.js, PHP, or Python. You can also run PowerShell and other scripts or executables as background services.

    üîÑ DevOps Integration

    Continuous integration and deployment with Azure DevOps, GitHub, BitBucket, Docker Hub, or Azure Container Registry. Promote updates through test and staging environments. Manage your apps using Azure PowerShell or cross-platform CLI.

    üåç Global Scale with High Availability

    Scale up or out manually or automatically. Host your apps anywhere in Microsoft's global datacenter infrastructure with built-in load balancing and Traffic Manager for high availability.

    üîó Connections to SaaS & On-Premises

    Connect to enterprise systems (SAP), SaaS services (Salesforce), and internet services (Facebook). Access on-premises data using Hybrid Connections and Azure Virtual Networks.

    üîí Security and Compliance

    App Service is ISO, SOC, and PCI compliant. Authenticate users with Microsoft Entra ID, Google, Facebook, X, or Microsoft account. Create IP address restrictions and manage service identities.

    üì± Application Templates

    Choose from an extensive list of application templates in the Azure Marketplace, such as WordPress, Joomla, and Drupal.

    üíª Visual Studio and VS Code Integration

    Dedicated tools in Visual Studio and Visual Studio Code streamline creating, deploying, and debugging.

    üîå API and Mobile Features

    App Service provides turn-key CORS support for RESTful API scenarios. Built-in authentication, offline data sync, push notifications, and more for mobile app scenarios.

    ‚ö° Serverless Code

    Run a code snippet or script on-demand without having to explicitly provision or manage infrastructure, and pay only for the compute time your code actually uses.

    üíé App Service Plans

    plans pricing tiers scaling

    An App Service plan defines a set of compute resources for a web app to run. These compute resources are analogous to the server farm in conventional web hosting. One or more apps can be configured to run on the same computing resources (or in the same App Service plan).

    üìä Pricing Tiers

    Tier Use Case Features Scale Limit
    Free (F1) Development & Testing Shared compute, 60 min/day, No custom domains N/A
    Shared (D1) Dev/Test with custom domain Shared compute, 240 min/day, Custom domains N/A
    Basic (B1, B2, B3) Low-traffic apps Dedicated compute, Manual scale, Custom domain + SSL Up to 3 instances
    Standard (S1, S2, S3) Production workloads Auto-scale, Staging slots (5), Daily backups Up to 10 instances
    Premium (P1v3, P2v3, P3v3) High-performance Enhanced performance, More slots (20), VNet integration Up to 30 instances
    Isolated (I1v2, I2v2, I3v2) High security & isolation App Service Environment, Network isolation, Max scale Up to 100 instances
    üí° Plan Sharing
    Multiple apps can share the same App Service plan, which means they share the same compute resources. This is cost-effective but requires careful capacity planning.

    ‚öñÔ∏è Scaling Options

    Scale Up (Vertical Scaling)

    Get more CPU, memory, disk space, and extra features like dedicated VMs, custom domains and certificates, staging slots, autoscaling, and more.

    Scale Out (Horizontal Scaling)

    Increase the number of VM instances that run your app. Standard tier allows up to 10 instances, Premium allows up to 30, and Isolated allows up to 100.

    Autoscale Configuration

    # Configure autoscale based on CPU

    üé≠ Deployment Slots

    slots staging swap blue-green

    Deployment slots are live apps with their own host names. App content and configurations elements can be swapped between two deployment slots, including the production slot.

    üéØ Benefits of Deployment Slots

    • Zero-downtime deployment - Swap staging to production without downtime
    • Warm-up before swap - All instances are warmed up before being swapped into production
    • Easy rollback - If issues arise, swap back to the previous version instantly
    • A/B testing - Route a percentage of traffic to different slots
    • Staged validation - Test in production-like environment before going live

    • Network Isolation - Deployed into your Azure Virtual Network subnet
    • High Scale - Support for up to 200 total App Service plan instances
    • Single Tenant - Dedicated to your subscription only
    • ILB or External - Internal Load Balancer or internet-facing endpoint
    • WAF Integration - Deploy behind Application Gateway with WAF
    • Compliance - Meets regulatory compliance requirements for isolation

    üîí Authentication & Security

    security authentication key-vault ssl

    Azure App Service provides comprehensive security features including built-in authentication, Key Vault integration, managed identities, and SSL/TLS certificate management.

    üîë Key Vault Integration

    Reference secrets from Azure Key Vault directly in your app settings using special syntax.

    # Enable managed identity
az webapp identity assign \
  --name MyWebApp \
  --resource-group MyResourceGroup

# Grant access to Key Vault
az keyvault set-policy \
  --name MyKeyVault \
  --object-id  \
  --secret-permissions get list

# Reference Key Vault secret in app setting
az webapp config appsettings set \
  --name MyWebApp \
  --resource-group MyResourceGroup \
  --settings \
    DB_PASSWORD="@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/DbPassword/)"

    Key Vault References in Code

    // Settings are automatically resolved from Key Vault
var password = Configuration["DB_PASSWORD"]; // Retrieved from Key Vault

// Check if reference was resolved
var isKeyVaultReference = Configuration["DB_PASSWORD"].StartsWith("@Microsoft.KeyVault");

    üÜî Managed Identities

    Managed identities provide an automatically managed identity in Microsoft Entra ID for applications to use when connecting to resources that support Microsoft Entra authentication.

    Types of Managed Identities

    • System-assigned - Tied to the lifecycle of the app service, deleted when app is deleted
    • User-assigned - Standalone resource that can be assigned to multiple apps
    # Enable system-assigned identity
az webapp identity assign \
  --name MyWebApp \
  --resource-group MyResourceGroup

# Create user-assigned identity
az identity create \
  --name MyUserIdentity \
  --resource-group MyResourceGroup

# Assign user-assigned identity to app
az webapp identity assign \
  --name MyWebApp \
  --resource-group MyResourceGroup \
  --identities /subscriptions//resourcegroups/MyResourceGroup/providers/Microsoft.ManagedIdentity/userAssignedIdentities/MyUserIdentity

    Using Managed Identity in Code

    // Access Azure SQL using managed identity
using Azure.Identity;
using Azure.Security.KeyVault.Secrets;

// Create client using DefaultAzureCredential (uses managed identity in Azure)
var client = new SecretClient(
    new Uri("https://myvault.vault.azure.net/"),
    new DefaultAzureCredential());

// Retrieve secret
KeyVaultSecret secret = await client.GetSecretAsync("DbPassword");
string password = secret.Value;

    Access on-premises resources without requiring a VPN. Each Hybrid Connection correlates to a single TCP host:port combination.

    üîí Private Endpoints

    Use Azure Private Link to access your web app privately from your VNet, eliminating exposure from the public internet.

    üåê Networking Capabilities by Tier
    ‚Ä¢ Basic/Free/Shared: IP Restrictions only

    ‚Ä¢ Premium: + VNet Integration, Private Endpoints

    ResponseWriter = async (context, report) =>
    {
        context.Response.ContentType = "application/json";

            status = report.Status.ToString(),
            checks = report.Entries.Select(e => new

            })

    üíæ Backup and Restore

    Available in Standard tier and higher. Backup up to 10 GB of app and database content.

    # Configure automated backup
az webapp config backup update \
  --resource-group MyResourceGroup \
  --webapp-name MyWebApp \
  --container-url "https://mybackupstorage.blob.core.windows.net/backups?" \
  --frequency 1d \
  --retain-one true \
  --retention 30

# Create on-demand backup
az webapp config backup create \
  --resource-group MyResourceGroup \
  --webapp-name MyWebApp \
  --container-url "https://mybackupstorage.blob.core.windows.net/backups?" \
  --backup-name manual-backup-$(date +%Y%m%d%H%M%S)

# Restore from backup
az webapp config backup restore \
  --resource-group MyResourceGroup \
  --webapp-name MyWebApp \
  --container-url "https://mybackupstorage.blob.core.windows.net/backups?" \
  --backup-name backup-name \
  --overwrite

    What Gets Backed Up

    • App configuration
    • File content
    • Azure SQL Database or Azure Database for MySQL (if configured)
    • Custom domains & SSL certificates

    ‚ú® Best Practices

    best-practices recommendations architecture

    Follow these best practices to build secure, scalable, and reliable applications on Azure App Service.

    üèóÔ∏è Architecture & Design

    • Separate environments - Use deployment slots for dev, staging, and production
    • Stateless applications - Store session state in Redis or Azure Storage
    • Connection pooling - Reuse database connections to avoid exhaustion
    • Asynchronous patterns - Use async/await to maximize thread pool efficiency
    • Microservices readiness - Design apps to be independently deployable

    üîí Security Best Practices

    • Use managed identities - Eliminate credentials from code and configuration
    • Store secrets in Key Vault - Never hardcode sensitive data
    • Enable HTTPS only - Enforce TLS 1.2 or higher
    • Implement Easy Auth - Use built-in authentication when possible
    • Network isolation - Use VNet integration and Private Endpoints for sensitive workloads
    • Regular updates - Keep runtime versions and frameworks current

    ‚ö° Performance Optimization

    • Enable Always On - Prevent cold starts for production apps (Standard tier+)
    • Configure autoscale - Scale based on metrics, not schedules
    • Use CDN - Offload static content to Azure CDN or Azure Front Door
    • Optimize database queries - Use connection pooling and proper indexing
    • Enable response compression - Reduce payload size for faster transfers
    • Implement caching - Use Azure Cache for Redis for frequently accessed data

    üîÑ Deployment Best Practices

    • Use CI/CD pipelines - Automate builds and deployments with GitHub Actions or Azure DevOps
    • Slot swaps for zero downtime - Test in staging, then swap to production
    • Slot-specific settings - Mark environment-specific settings as slot settings
    • Pre-swap validation - Use auto-swap with warm-up to catch issues before production
    • Rollback strategy - Keep previous deployment ready for instant rollback
    • Blue-green deployments - Validate in parallel before switching traffic

    üìä Monitoring & Operations

    • Enable Application Insights - Monitor performance, failures, and dependencies
    • Configure alerts - Set up proactive alerts for critical metrics
    • Implement health checks - Auto-heal unhealthy instances
    • Diagnostic logging - Enable appropriate logging levels for troubleshooting
    • Regular backups - Schedule automated backups (Standard tier+)
    • Cost monitoring - Track resource usage and optimize App Service Plans

    üí∞ Cost Optimization

    • Right-size your plan - Start with Basic/Standard, scale up only when needed
    • Share App Service Plans - Run multiple apps on same plan when possible
    • Use autoscale rules - Scale down during off-peak hours
    • Stop dev/test apps - Stop non-production apps when not in use
    • Use free managed certificates - Avoid paying for SSL certificates
    • Reserved instances - Purchase Reserved Capacity for predictable workloads (Premium/Isolated)

    üåê High Availability & Disaster Recovery

    • Zone redundancy - Deploy across availability zones (Premium v3, ASE v3)
    • Multi-region deployment - Use Azure Front Door or Traffic Manager
    • Health probes - Configure proper health endpoints for load balancers
    • Regular testing - Test backup restoration and failover procedures
    • Data replication - Use geo-replication for databases
    • Documented runbooks - Maintain clear disaster recovery procedures
    ‚≠ê Golden Rules
    1. Never hardcode secrets - use Key Vault + Managed Identity
    2. Always use deployment slots for production deployments
    3. Enable Application Insights from day one
    4. Configure autoscaling before you need it
    5. Test disaster recovery procedures regularly
