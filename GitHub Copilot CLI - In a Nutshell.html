    Available Commands

    Slash Commands

    Command Purpose Example
    /model Change the AI model Select from available models
    /mcp Manage MCP servers View and configure MCP servers
    /feedback Provide feedback Submit survey, bug report, or feature request
    /help Show available commands Display help information
    /exit or /quit End the session Exit interactive mode

    Tool Approval in Interactive Mode

    When Copilot needs to use a tool (execute commands, modify files), it will ask for your approval:

    Approval Options

    1. Yes
2. Yes, and approve TOOL for the rest of the running session
3. No, and tell Copilot what to do differently (Esc)

    Option 1: One-time Approval

    • Approves only this specific action
    • Copilot will ask again for the same tool later
    • Safest option for sensitive operations

    Option 2: Session-wide Approval

    • Approves this tool for the entire session
    • No more prompts for this tool until session ends
    • Convenient for repetitive tasks
    • ‚ö†Ô∏è Security Note: If you approve rm for one file, Copilot can use any rm command

    Option 3: Reject and Redirect

    • Cancels the current action
    • Allows you to provide alternative instructions
    • Useful when Copilot misunderstood your intent

    Session Management

    Pausing and Resuming

    You can pause and resume sessions:

    Pause Session

    Ctrl+C  # Pauses the current operation
/exit   # Saves and exits the session

    Resume Session

    Sessions are automatically saved. Start copilot again to continue where you left off:

    copilot

    Session History

    • Previous prompts and responses are remembered
    • Context is maintained across session resumes
    • Session data is stored in ~/.copilot/sessions/

    ‚ö° Programmatic Mode

    programmatic command script automation

    Programmatic mode allows you to execute Copilot commands directly from the command line without starting an interactive session. This is ideal for automation, scripting, and CI/CD pipelines.

    Basic Syntax

    copilot -p "YOUR_PROMPT_HERE"

    Or using the long form:

    copilot --prompt "YOUR_PROMPT_HERE"

    Command-Line Options

    Option Description Example
    -p, --prompt Specify the prompt -p "List files"
    --allow-all-tools Auto-approve all tools Use with caution
    --allow-tool Auto-approve specific tool --allow-tool 'shell(git)'
    --deny-tool Block specific tool --deny-tool 'shell(rm)'
    --version Show version copilot --version
    --help Show help information copilot --help

    Tool Approval in Programmatic Mode

    In programmatic mode, you must specify tool permissions via command-line options since there's no interactive approval.

    Allow All Tools (Dangerous)

    copilot -p "Revert the last commit" --allow-all-tools
    ‚ö†Ô∏è Security Warning
    Using --allow-all-tools gives Copilot unrestricted access to run any command without approval. Only use this in trusted, isolated environments like containers or VMs.

    Allow Specific Tools

    Allow Shell Commands

    # Allow all shell commands
copilot -p "Show recent commits" --allow-tool 'shell'

# Allow specific command
copilot -p "Show git status" --allow-tool 'shell(git)'

# Allow git subcommand
copilot -p "Create a new branch" --allow-tool 'shell(git branch)'

    Allow File Write Operations

    copilot -p "Create a README file" --allow-tool 'write'

    Allow MCP Server Tools

    # Allow all tools from an MCP server
copilot -p "Query database" --allow-tool 'My-MCP-Server'

# Allow specific tool from MCP server
copilot -p "Run query" --allow-tool 'My-MCP-Server(query_tool)'

    Deny Specific Tools

    Use --deny-tool to block dangerous operations:

    # Allow all but deny dangerous commands
copilot -p "Clean up project" \
  --allow-all-tools \
  --deny-tool 'shell(rm)' \
  --deny-tool 'shell(git push)'

    Combining Allow and Deny

    # Allow git except push
copilot -p "Manage repository" \
  --allow-tool 'shell(git)' \
  --deny-tool 'shell(git push)'

    Note: --deny-tool takes precedence over --allow-tool and --allow-all-tools.

    Piping Input

    You can pipe script output to Copilot:

    echo ./my-script.sh | copilot

    Or pipe from a file:

    cat prompt.txt | copilot

    4. Quick Queries

    5. Scheduled Tasks

    Output Handling

    Redirect to File

    copilot -p "Create a Python hello world script" --allow-tool 'write' > hello.py

    Append to File

    copilot -p "Add error handling tips" >> guidelines.md

    Pipe to Another Command

    copilot -p "List large files in this repo" --allow-tool 'shell' | grep ".mp4"

    Capture in Variable

    result=$(copilot -p "Count lines of code in src/" --allow-tool 'shell')

    Error Handling

  echo "Success"

  echo "Failed"

    Timeout Control

    # Add timeout for long-running operations

    Best Practices for Programmatic Mode

    ‚úÖ Security Best Practices

    • Use specific tool permissions: Prefer --allow-tool over --allow-all-tools
    • Deny dangerous operations: Always block rm, git push in production
    • Run in isolated environments: Use containers or VMs for automation
    • Validate output: Don't blindly execute Copilot-generated commands
    • Use read-only mode when possible: Avoid --allow-tool 'write' for queries

    ‚úÖ Scripting Best Practices

    • Be specific in prompts: Clear prompts produce better results
    • Handle errors gracefully: Check exit codes and validate output
    • Log operations: Keep audit trail of what Copilot does
    • Test before production: Validate scripts in development first
    • Document tool permissions: Comment why each tool is allowed

    ‚ùå Things to Avoid

    • Don't use in production without testing: Always validate in dev/staging first
    • Don't grant excessive permissions: Follow principle of least privilege
    • Don't ignore security warnings: Take --allow-all-tools warnings seriously
    • Don't run without rate limiting: Respect API quotas and premium request limits

    ü™ù Hooks

    hooks validation automation

    Hooks allow you to execute custom shell commands at key points during agent execution. This enables validation, logging, security scanning, workflow automation, and custom checks before or after Copilot performs actions.

    What Are Hooks?

    Hooks are custom shell scripts that run automatically at specific execution points:

    • Pre-execution hooks: Run before Copilot executes an action
    • Post-execution hooks: Run after Copilot completes an action
    • Validation hooks: Check conditions before proceeding
    • Notification hooks: Alert about completed actions
    üí° Use Cases
    Hooks are perfect for adding validation (lint checks, security scans), logging (audit trails), notifications (Slack alerts), and custom workflows (auto-formatting, doc generation).

    Hook Types

    Hook Type When It Runs Common Uses
    pre-execute Before any command execution Validation, permission checks
    post-execute After command completes Logging, notifications, cleanup
    pre-write Before writing files Backup, validation
    post-write After writing files Formatting, linting, testing
    pre-commit Before git commits Code quality checks
    post-commit After git commits Notifications, CI trigger

    ‚úÖ Trusted Directories

    trust whitelist access-control

    Trusted directories provide a security layer by explicitly defining which directories Copilot is allowed to operate in. This prevents accidental operations in sensitive locations like system directories or unintended project folders.

    Why Use Trusted Directories?

    • Prevent accidents: Avoid operations in wrong directories
    • Protect sensitive areas: Block access to system or production paths
    • Team standards: Ensure team members operate in approved locations
    • Compliance: Meet security requirements for access control
    • Peace of mind: Reduce anxiety about AI operations
    üí° How It Works
    When trusted directories are configured, Copilot will refuse to operate in directories outside the trusted list, requiring you to explicitly trust new locations before use.

    Configuring Trusted Directories

    Configuration File

    Edit ~/.copilot/config.json:

    {
  "trusted_directories": [
    "~/projects",
    "~/workspace",
    "/opt/company-projects"
  ]
}

    Trust Patterns

    Use wildcards for flexible matching:

    {
  "trusted_directories": [
    "~/projects/*",           // All subdirectories under projects
    "~/work/team-*",          // Any directory starting with team-
    "/home/*/safe-repos"      // safe-repos in any user's home
  ]
}

    Trust Levels

    Level Configuration Use Case
    No Trust Empty or no config Trust all directories (default)
    Specific Paths Exact directory paths Precise control over locations
    Pattern-Based Wildcards and patterns Flexible matching for multiple dirs
    Hierarchical Parent directory trust Trust entire project structure

    tools permissions approval

    Tool approval is a critical security mechanism that requires explicit permission before Copilot can execute commands, access external services, or perform potentially dangerous operations. This prevents unexpected or harmful actions.

    What Are Tools?

    In Copilot CLI context, "tools" refer to:

    • Shell commands: Any command-line executable (npm, git, docker, etc.)
    • MCP server functions: External service integrations
    • File operations: Reading, writing, deleting files
    • System operations: Network calls, service management

    Tool Approval Modes

    Mode Behavior Use Case
    Interactive (Default) Prompt before each tool use Safe for general use
    Allow All Grant permission to all tools Trusted scripts only
    Allow Specific Pre-approve certain tools Automated workflows
    Deny Specific Block certain tools Prevent dangerous operations
    Deny All Block all tool execution Read-only operations

    Granting Tool Permissions

    Interactive Mode (Default)

    Copilot will prompt when it needs to use a tool:

    copilot -p "Install dependencies"

Copilot wants to use: shell(npm install)
Allow this tool? (y/n/always): y

    Programmatic Mode - Allow Specific Tools

    # Allow specific tool
copilot -p "Build the project" --allow-tool 'shell(npm)'

# Allow multiple tools
copilot -p "Deploy app" --allow-tool 'shell(docker)' --allow-tool 'shell(kubectl)'

# Allow all shell commands
copilot -p "Setup environment" --allow-tool 'shell'

# Allow MCP server tools

    Deny Specific Tools

    # Block dangerous commands
copilot -p "Clean up files" --deny-tool 'shell(rm)' --deny-tool 'shell(dd)'

# Allow shell but deny specific commands
copilot -p "Run scripts" --allow-tool 'shell' --deny-tool 'shell(rm)' --deny-tool 'shell(shutdown)'

    Tool Categories

    Safe Tools (Low Risk)

    • shell(ls), shell(pwd), shell(whoami) - Information gathering
    • shell(cat), shell(less) - File reading
    • shell(git status), shell(git log) - Git info
    • shell(npm list), shell(pip list) - Package info

    Moderate Risk Tools

    • shell(git commit), shell(git push) - Version control
    • shell(npm install), shell(pip install) - Package installation
    • write - File writing
    • shell(mkdir), shell(touch) - File system creation

    High Risk Tools

    • shell(rm), shell(rmdir) - File deletion
    • shell(chmod), shell(chown) - Permission changes
    • shell(docker), shell(kubectl) - Container/cluster operations
    • shell(systemctl), shell(service) - Service management
    • Database MCP tools - Data modification

    Critical Risk Tools (Extreme Caution)

    • shell(dd) - Low-level disk operations
    • shell(mkfs) - Filesystem creation
    • shell(shutdown), shell(reboot) - System control
    • shell(iptables), shell(ufw) - Firewall changes
