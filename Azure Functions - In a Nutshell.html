    üìñ Overview

    intro overview basics

    Azure Functions is a serverless compute service that enables you to run event-driven code without having to explicitly provision or manage infrastructure. You can focus on writing code while Azure handles the scaling, infrastructure management, and operational concerns.

    üéØ Core Purpose
    Azure Functions allows you to run small pieces of code (functions) in response to events without worrying about infrastructure. Write your business logic, and Azure automatically handles scaling, availability, and resource management‚Äîpaying only for the compute time you consume.

    Key Benefits

    • ‚ö° Event-Driven Execution - Automatically triggered by various events (HTTP, timers, queues, etc.)
    • üöÄ Automatic Scaling - Scales automatically based on demand, from zero to thousands of instances
    • üí∞ Pay-per-Use - Only pay for actual execution time (Consumption plan)
    • üîß Multiple Languages - Support for C#, Java, JavaScript, TypeScript, Python, PowerShell
    • üîó Integrated Bindings - Easy integration with Azure services through declarative bindings
    • üìä Built-in Monitoring - Application Insights integration for telemetry and diagnostics

    Common Use Cases

    ‚è∞ Scheduled Tasks

    Run periodic tasks like data cleanup, report generation, or batch processing without managing servers. Perfect for cron-like jobs with timer triggers.

    üåê HTTP APIs & Webhooks

    Build serverless REST APIs and webhooks that scale automatically. Integrate with API Management for enterprise-grade API capabilities.

    üì® Event Processing

    React to events from Azure services (Blob Storage, Event Hubs, Service Bus, Cosmos DB) for real-time data processing and workflows.

    ü§ñ Backend for Mobile & Web Apps

    Create lightweight backend APIs for mobile and single-page applications with built-in authentication and authorization support.

    üîÑ Microservices & Integration

    Implement microservices architecture patterns or integrate systems using Azure Functions as glue code between services.

    üíæ Data Processing Pipelines

    Process streams of data for ETL operations, IoT telemetry, or real-time analytics with event-driven architecture.

    Why Choose Azure Functions?

    • Serverless - No infrastructure management, automatic scaling, high availability
    • Developer Productivity - Focus on code, not infrastructure; rich IDE support
    • Flexible Hosting - Multiple hosting plans (Consumption, Premium, Dedicated)
    • Rich Ecosystem - Extensions, bindings, and integrations with Azure services
    • Open Source Runtime - Deploy anywhere (Azure, on-premises, other clouds)
    üí° Serverless = Focus on Business Logic
    With Azure Functions, you write code that solves business problems without managing VMs, configuring load balancers, or worrying about OS patches. Azure handles all operational aspects, letting you focus on delivering value.

    1Ô∏è‚É£ Function App

    A Function App is the execution context and container for one or more functions. It provides:

    • üì¶ Deployment unit: Functions are deployed together as a single package
    • ‚öôÔ∏è Configuration management: Shared settings, connection strings, app settings
    • üîê Security context: Authentication, managed identities, secrets
    • üí∞ Billing boundary: Cost is tracked at the Function App level
    • üöÄ Scaling unit: All functions in the app scale together
    • üåç Runtime environment: Shared language runtime and dependencies

    2Ô∏è‚É£ Functions

    Individual Functions are the building blocks containing your business logic:

    • üìù Code files: Your application logic written in supported languages
    • üîå Trigger: Exactly one trigger that starts function execution
    • üîó Bindings: Zero or more input/output bindings for data integration
    • ‚ö° Stateless by default: Each invocation is independent (unless using Durable Functions)
    • üîÑ Reusable: Can be called multiple times concurrently

    4Ô∏è‚É£ Bindings

    Bindings are declarative connections to data sources, eliminating boilerplate code:

    Input Bindings

    • üì• Read data from external sources (databases, blob storage, etc.)
    • üîÑ Data is automatically deserialized and passed to your function
    • ‚ö° No need for SDK code to connect and read data

    Output Bindings

    • üì§ Write data to external destinations
    • üîÑ Data is automatically serialized and sent
    • ‚öôÔ∏è Multiple output bindings per function are supported

    Common Binding Types

    • Azure Storage: Blobs, Queues, Tables
    • Cosmos DB: Read/write documents
    • Service Bus: Send messages
    • Event Hubs: Send events
    • HTTP: Return HTTP responses
    • SendGrid: Send emails
    • Twilio: Send SMS
    • SignalR: Real-time messaging
    üí° Bindings = Less Code
    Bindings significantly reduce the amount of code you need to write. Instead of manually connecting to databases or storage accounts, you define bindings declaratively, and Azure Functions handles all the connection management, retry logic, and data transformation.

    5Ô∏è‚É£ Azure Functions Runtime

    The Functions Runtime is the engine that executes your functions:

    • üîß Open Source: Available on GitHub
    • üåç Portable: Run anywhere (Azure, on-premises, other clouds)
    • üîÑ Version Management: Runtime versions (v1, v2, v3, v4) with long-term support
    • üì¶ Language Support: Multiple language workers (C#, Java, Python, Node.js, etc.)
    • ‚ö° Execution Management: Handles scaling, lifecycle, logging, monitoring

    Architecture Patterns

    Simple Function Pattern

    // HTTP trigger with Queue output binding
[FunctionName("ProcessOrder")]
public static async Task<IActionResult> Run(
    [HttpTrigger(AuthorizationLevel.Function, "post")] HttpRequest req,
    [Queue("orders")] IAsyncCollector<string> outputQueue,
    ILogger log)
{
    string requestBody = await new StreamReader(req.Body).ReadToEndAsync();
    await outputQueue.AddAsync(requestBody);
    return new OkResult();
}

    Fan-Out/Fan-In Pattern

    // Queue trigger processing with parallel execution
[FunctionName("ProcessBatch")]
public static async Task Run(
    [QueueTrigger("batch-items")] string item,
    [Blob("results/{rand-guid}.json")] Stream outputBlob,
    ILogger log)
{
    var result = await ProcessItemAsync(item);
    await outputBlob.WriteAsync(result);
}
    üèóÔ∏è Best Practice
    Keep functions small and focused. Each function should do one thing well. Use bindings to minimize code, and leverage Durable Functions for complex workflows requiring state management or orchestration.

    üîå Triggers & Bindings

    triggers bindings events

    Triggers and bindings are powerful declarative features that minimize the code needed to integrate with Azure services and external systems.

    Understanding Triggers

    A trigger defines how a function is invoked. Each function must have exactly one trigger.

    HTTP Trigger

    Invoke functions via HTTP requests‚Äîperfect for APIs and webhooks.

    [FunctionName("HttpExample")]
public static async Task<IActionResult> Run(
    [HttpTrigger(AuthorizationLevel.Function, "get", "post")] HttpRequest req,
    ILogger log)
{
    string name = req.Query["name"];
    return new OkObjectResult($"Hello, {name}");
}

    Timer Trigger

    Schedule function execution using cron expressions.

    // Runs every day at 2:00 AM
[FunctionName("DailyCleanup")]
public static void Run(
    [TimerTrigger("0 0 2 * * *")] TimerInfo myTimer,
    ILogger log)
{
    log.LogInformation($"Cleanup executed at: {DateTime.Now}");
}

    Cron Expression Examples

    Expression Description
    0 */5 * * * * Every 5 minutes
    0 0 * * * * Every hour
    0 0 9 * * 1-5 9 AM weekdays
    0 0 0 * * * Daily at midnight

    Queue Trigger

    Process messages from Azure Storage Queues or Service Bus Queues.

    [FunctionName("QueueProcessor")]
public static void Run(
    [QueueTrigger("orders", Connection = "AzureWebJobsStorage")] string myQueueItem,
    ILogger log)
{
    log.LogInformation($"Processing order: {myQueueItem}");
}

    Blob Trigger

    React to new or updated blobs in Azure Blob Storage.

    [FunctionName("ImageProcessor")]
public static void Run(
    [BlobTrigger("images/{name}", Connection = "AzureWebJobsStorage")] Stream imageBlob,
    string name,
    ILogger log)
{
    log.LogInformation($"Processing blob: {name}, Size: {imageBlob.Length} bytes");
}

    Cosmos DB Trigger

    React to changes in Cosmos DB using the change feed.

    [FunctionName("CosmosDBProcessor")]
public static void Run(
    [CosmosDBTrigger(
        databaseName: "MyDatabase",
        containerName: "MyContainer",
        Connection = "CosmosDBConnection",
        LeaseContainerName = "leases")] IReadOnlyList<Document> documents,
    ILogger log)
{
    foreach (var doc in documents)
    {
        log.LogInformation($"Document modified: {doc.Id}");
    }
}

    Event Hubs Trigger

    Process high-throughput event streams from Azure Event Hubs.

    [FunctionName("EventHubProcessor")]
public static void Run(
    [EventHubTrigger("telemetry", Connection = "EventHubConnection")] EventData[] events,
    ILogger log)
{
    foreach (var evt in events)
    {
        log.LogInformation($"Event: {Encoding.UTF8.GetString(evt.EventBody)}");
    }
}

    ‚è±Ô∏è Durable Functions

    durable orchestration stateful workflows

    Durable Functions is an extension that enables you to write stateful workflows in a serverless environment. It manages state, checkpoints, and restarts automatically.

    Why Durable Functions?

    • üîÑ Stateful workflows: Chain functions together with state management
    • ‚è±Ô∏è Long-running processes: Handle operations that take hours or days
    • üîÅ Automatic retries: Built-in retry logic with exponential backoff
    • üíæ Checkpoint and replay: Automatic progress tracking and recovery
    • ‚ö° Parallel execution: Fan-out/fan-in patterns made easy
    • üë§ Human interaction: Wait for external events or approvals

    Key Concepts

    Orchestrator Functions

    Orchestrator functions define the workflow logic and coordinate other functions:

    [FunctionName("OrderOrchestrator")]
public static async Task<string> RunOrchestrator(
    [OrchestrationTrigger] IDurableOrchestrationContext context)
{
    var order = context.GetInput<Order>();
    
    // Sequential execution
    await context.CallActivityAsync("ProcessPayment", order);
    await context.CallActivityAsync("UpdateInventory", order);
    await context.CallActivityAsync("ShipOrder", order);
    
    return "Order completed";
}

    Activity Functions

    Activity functions contain the actual work to be performed:

    [FunctionName("ProcessPayment")]
public static async Task ProcessPayment([ActivityTrigger] Order order, ILogger log)
{
    log.LogInformation($"Processing payment for order {order.Id}");
    await PaymentService.ChargeAsync(order.Amount);
}

    Client Functions

    Start and monitor orchestrations:

    [FunctionName("StartOrchestration")]
public static async Task<IActionResult> Start(
    [HttpTrigger(AuthorizationLevel.Function, "post")] HttpRequest req,
    [DurableClient] IDurableOrchestrationClient client)
{
    var order = await req.ReadAsAsync<Order>();
    string instanceId = await client.StartNewAsync("OrderOrchestrator", order);
    
    return client.CreateCheckStatusResponse(req, instanceId);
}

    1. Function Chaining

    Execute functions in a sequence:

    2. Fan-Out/Fan-In

    Execute multiple functions in parallel and aggregate results:

    %%{ init: { "theme": "base", "themeVariables": { "fontFamily": "Segoe UI" } } }%% flowchart TB Start[Start] --> Split{Fan-Out} Split --> F1[Task 1] Split --> F2[Task 2] Split --> F3[Task 3] F1 --> Merge{Fan-In} F2 --> Merge F3 --> Merge Merge --> End[Aggregate Results] style Start fill:#10b981,color:#fff style End fill:#3b82f6,color:#fff 
    public static async Task<long> Run(IDurableOrchestrationContext context)
{
    var tasks = new List<Task<int>>();
    for (int i = 0; i < 100; i++)
    {
        tasks.Add(context.CallActivityAsync<int>("ProcessItem", i));
    }
    
    var results = await Task.WhenAll(tasks);
    return results.Sum();
}

    4. Monitoring Pattern

    Periodic monitoring with conditional logic:

    public static async Task Run(IDurableOrchestrationContext context)
{
    var jobId = context.GetInput<string>();
    DateTime expiryTime = context.CurrentUtcDateTime.AddMinutes(30);
    
    while (context.CurrentUtcDateTime < expiryTime)
    {
        var status = await context.CallActivityAsync<string>("CheckJobStatus", jobId);
        
        if (status == "Completed")
            break;
        
        // Wait 30 seconds before checking again
        await context.CreateTimer(context.CurrentUtcDateTime.AddSeconds(30), CancellationToken.None);
    }
}

    5. Human Interaction

    Wait for external events or approvals:

    public static async Task<bool> Run(IDurableOrchestrationContext context)
{
    var approvalTask = context.WaitForExternalEvent<bool>("ApprovalEvent");
    var timeoutTask = context.CreateTimer(context.CurrentUtcDateTime.AddDays(3), CancellationToken.None);
    
    var winner = await Task.WhenAny(approvalTask, timeoutTask);
    
    if (winner == approvalTask)
        return await approvalTask;
    else
        return false; // Timeout
}

    var counter = context.GetInput<int>();
    
    await context.CallActivityAsync("DoWork", counter);
    
    context.ContinueAsNew(counter + 1); // Restart with new state
}

    Sub-Orchestrations

    Call orchestrations from within orchestrations:

    public static async Task Run(IDurableOrchestrationContext context)
{
    var results = new List<string>();
    
    foreach (var item in items)
    {
        var result = await context.CallSubOrchestratorAsync<string>(
            "ItemProcessor", 
            item);
        results.Add(result);
    }
}

    Best Practices

    • ‚úÖ Keep orchestrators deterministic: No random numbers, DateTime.Now, or HTTP calls
    • ‚úÖ Use activity functions for I/O: Database calls, HTTP requests, file operations
    • ‚úÖ Handle timeouts: Implement timeout patterns for long-running operations
    • ‚úÖ Version orchestrators: Use versioning for breaking changes
    • ‚úÖ Monitor instances: Track status and clean up completed instances
    ‚ö° Durable Functions = Stateful Serverless
    Durable Functions brings the power of stateful workflows to serverless. Write complex business processes as code without managing state infrastructure, queues, or coordination logic.

    Detailed Plan Features

    üü¢ Consumption Plan

    • Automatic scaling: Platform automatically adds/removes instances
    • Pricing: Pay only for execution time + memory consumption
    • Free grant: 1M executions + 400,000 GB-s per month
    • Cold starts: First request after idle period may be slow (2-10 seconds)
    • Limitations: No VNet integration, max 10 min timeout, limited instance size

    üîµ Premium Plan (Elastic Premium)

    • Pre-warmed instances: Always-on workers eliminate cold starts
    • VNet integration: Access private resources securely
    • Unlimited execution duration: Long-running functions supported
    • Enhanced hardware: Faster CPUs, more memory (up to 14 GB)
    • Pricing: Based on number and size of instances + executions
    • SKUs: EP1, EP2, EP3 (increasing compute power)

    • Predictable pricing: Fixed monthly cost regardless of usage
    • Always on: No cold starts
    • Shared resources: Run alongside web apps on same plan
    • Full control: Manual or auto-scale rules
    • Use existing plan: Leverage underutilized App Service plan capacity

    • Idle instances removed after functions stop executing

    ‚ú® Best Practices

    best-practices security reliability

    Function Design Principles

    • ‚úÖ Single Responsibility: Each function should do one thing well
    • ‚úÖ Stateless by design: Don't rely on in-memory state
    • ‚úÖ Idempotent operations: Safe to retry without side effects
    • ‚úÖ Small and focused: Keep execution time short (seconds, not minutes)
    • ‚úÖ Defensive coding: Validate inputs, handle errors gracefully

    Security

    Authentication & Authorization

    • ‚úÖ Use Azure AD/Entra ID for authentication (not anonymous)
    • ‚úÖ Implement function-level authorization with access keys
    • ‚úÖ Use managed identities to access Azure resources
    • ‚úÖ Validate JWT tokens for API endpoints
    • ‚úÖ Apply least privilege to service principals and identities

    Secrets Management

    • ‚úÖ Store secrets in Azure Key Vault (not in code or app settings)
    • ‚úÖ Use Key Vault references in app settings
    • ‚úÖ Rotate keys and secrets regularly
    • ‚úÖ Never commit secrets to source control
    // Application Settings - Key Vault reference\n{\n  \"MySecret\": \"@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/MySecret/)\",\n  \"CosmosDBConnection\": \"@Microsoft.KeyVault(SecretUri=https://myvault.vault.azure.net/secrets/CosmosDB/)\"\n}

    Network Security

    • ‚úÖ Use VNet integration for outbound connectivity (Premium/Dedicated)
    • ‚úÖ Enable private endpoints for inbound isolation
    • ‚úÖ Configure IP restrictions for HTTP triggers
    • ‚úÖ Use Azure Front Door or API Management as a gateway

    ‚úÖ Performance Optimization

    Connection Management

    • ‚úÖ Reuse connections: Initialize clients outside handler function
    • ‚úÖ Use connection pooling for databases and HTTP clients
    • ‚úÖ Static/singleton pattern: Share expensive resources across invocations
    // ‚úÖ Good - Reuse HttpClient\npublic class MyFunction\n{\n    private static readonly HttpClient httpClient = new HttpClient();\n    \n    [FunctionName(\"MyFunction\")]\n    public static async Task Run(/* bindings */)\n    {\n        var response = await httpClient.GetAsync(\"https://api.example.com\");\n    }\n}\n\n// ‚ùå Bad - Creates new HttpClient each time\npublic static async Task Run(/* bindings */)\n{\n    using var httpClient = new HttpClient(); // DON'T DO THIS\n    var response = await httpClient.GetAsync(\"https://api.example.com\");\n}

    Async Patterns

    • ‚úÖ Use async/await for all I/O operations
    • ‚úÖ Never block on async operations (.Result or .Wait())
    • ‚úÖ Use ConfigureAwait(false) in library code

    Memory Management

    • ‚úÖ Dispose of resources properly (use using statements)
    • ‚úÖ Avoid large in-memory objects
    • ‚úÖ Stream large payloads instead of loading entirely
    • ‚úÖ Monitor memory usage in Application Insights

    ‚úÖ Reliability & Error Handling

    Retry Logic

    • ‚úÖ Use exponential backoff for transient failures
    • ‚úÖ Implement circuit breaker pattern for external dependencies
    • ‚úÖ Configure retry policies in host.json for triggers
    // host.json - Retry configuration\n{\n  \"version\": \"2.0\",\n  \"extensions\": {\n    \"serviceBus\": {\n      \"prefetchCount\": 100,\n      \"messageHandlerOptions\": {\n        \"maxConcurrentCalls\": 32,\n        \"maxAutoRenewDuration\": \"00:05:00\"\n      }\n    },\n    \"queues\": {\n      \"maxPollingInterval\": \"00:00:02\",\n      \"visibilityTimeout\" : \"00:00:30\",\n      \"batchSize\": 16,\n      \"maxDequeueCount\": 5\n    }\n  }\n}

    Error Logging

    • ‚úÖ Use ILogger for structured logging
    • ‚úÖ Log exceptions with context and correlation IDs
    • ‚úÖ Avoid logging sensitive data (PII)
    • ‚úÖ Set up alerts for critical errors

    Dead Letter Queues

    • ‚úÖ Configure poison message handling
    • ‚úÖ Use dead letter queues for failed messages
    • ‚úÖ Monitor and process dead letter queues

    Monitoring & Observability

    • ‚úÖ Enable Application Insights for all functions
    • ‚úÖ Use custom metrics and events for business logic tracking
    • ‚úÖ Configure availability tests for critical endpoints
    • ‚úÖ Set up alerts for failures, latency, and anomalies
    • ‚úÖ Use dependency tracking to identify bottlenecks
    • ‚úÖ Monitor scaling metrics (instance count, CPU, memory)

    Common Anti-Patterns

    • ‚ùå Creating HttpClient instances per invocation (socket exhaustion)
    • ‚ùå Blocking async operations with .Result or .Wait()
    • ‚ùå Storing state in static variables (unreliable in serverless)
    • ‚ùå Long-running operations in Consumption plan (use Premium/Durable)
    • ‚ùå Synchronous processing of high-volume queues (use batching)
    • ‚ùå Not handling exceptions (functions fail silently)
    • ‚ùå Hardcoding connection strings and secrets
    • ‚ùå No monitoring or logging (blind to failures)
    • ‚ùå Using anonymous HTTP auth in production
    • ‚ùå Not setting maxConcurrentCalls for event-driven triggers
    üéØ Production Readiness Checklist
    1. ‚úÖ Security: Managed identities + Key Vault + Azure AD auth
    2. ‚úÖ Performance: Connection reuse + async patterns + batching
    3. ‚úÖ Reliability: Retry logic + error handling + dead letter queues
    4. ‚úÖ Monitoring: Application Insights + alerts + custom metrics
    5. ‚úÖ Cost: Right hosting plan + optimized execution + cleanup
    6. ‚úÖ DevOps: CI/CD + IaC + deployment slots + testing
