    What is BMad Method?

    intro, overview, scales

    BMad Method (Breakthrough Method for Agile AI-Driven Development) is an AI-driven agile development methodology that scales from bug fixes to enterprise systems. Distributed as an NPM package, it provides 19 specialized AI agents and 50+ structured workflows across 3 adaptive tracks that automatically adjust to project complexity‚Äîdelivering planning in minutes, not hours or days.

    Scale-Adaptive Intelligence

    methodology, workflow-routing

    BMad Method automatically adjusts planning depth and workflow complexity based on project scope through a 3-track system:

    Three Planning Tracks

    ‚ö° Quick Flow Track

    For bug fixes and small features (1-15 stories) - Tech spec only - < 5 minutes
    • Scope: 1-15 stories, bug fixes, small enhancements
    • Planning Time: < 5 minutes
    • Artifacts: Tech spec only
    • Workflows: *tech-spec ‚Üí *quick-dev
    • Example: Fix authentication bug, add export feature

    üìã BMad Method Track

    For products and platforms (10-50+ stories) - PRD + Architecture + UX - < 15 minutes
    • Scope: 10-50+ stories, complete products
    • Planning Time: < 15 minutes
    • Artifacts: PRD, Architecture, UX Design, Epics
    • Workflows: *create-prd ‚Üí *create-architecture ‚Üí *create-ux-design ‚Üí *create-epics-and-stories
    • Example: E-commerce platform, SaaS product

    üè¢ Enterprise Track

    For compliance-heavy systems (30+ stories) - Full governance suite - < 30 minutes
    • Scope: 30+ stories, enterprise systems, compliance requirements
    • Planning Time: < 30 minutes
    • Artifacts: Product Brief, Research, PRD, Architecture, Security, DevOps, Compliance
    • Workflows: Full governance with *product-brief, *research, security workflows
    • Example: Banking system, healthcare platform

    Automatic Track Detection

    The *workflow-init workflow performs intelligent detection by analyzing:

    • üìÅ Existing project structure (greenfield vs brownfield)
    • üìÑ Existing artifacts (PRD, architecture, sprint-status.yaml)
    • üéØ Project goal complexity
    • üë• Team size and compliance requirements

    Core Components

    1. Unified .bmad/ Directory

    • Single hidden folder replacing v4's multiple directories (.bmad-core, .bmad-method, etc.)
    • Module-first organization: core/, bmm/, bmb/, cis/, _cfg/
    • Configuration hub (_cfg/): manifests, customizations, and tracking files
    • Update-safe design: User customizations survive framework updates
    • Intelligent detection: Auto-discovers installations via manifest.yaml

    2. C.O.R.E. Framework (BMad Core)

    Collaboration Optimized Reflection Engine
    • Collaboration: Agent orchestration with specialized personas
    • Optimized: Module system + workflow engine for refined processes
    • Reflection: discover_inputs protocol for strategic context discovery
    • Engine: Framework orchestrating agents, workflows, and IDEs

    3. Module System

    • BMM (BMad Method): 12 agents, 34 workflows across 4-phase agile methodology
    • BMB (BMad Builder): Tools for creating custom agents, workflows, and modules
    • CIS (Creative Intelligence Suite): 5 creative facilitation workflows
    • Pluggable architecture: Community modules for legal, medical, finance, education domains

    4. Agent Compilation Pipeline

    • YAML‚ÜíXML transformation: Agent definitions compiled to IDE-ready formats
    • "Write once, deploy everywhere": Single .agent.yaml generates 10+ IDE formats
    • Polymorphic handlers: IdeManager dynamically loads IDE-specific setup
    • Multi-IDE support: Claude Code, Cursor, Windsurf, Codex, Qwen, GitHub Copilot, and more
    • Customization deep-merge: User overrides (*.customize.yaml) applied at compile time

    5. Scale-Adaptive Intelligence

    • Quick Flow (Level 0-1): 1-15 stories, tech spec only, < 5 minutes
    • BMad Method (Level 2-3): 10-50+ stories, PRD + Architecture + UX, < 15 minutes
    • Enterprise (Level 3-4): 30+ stories, full planning suite + security, < 30 minutes
    • Automatic detection: *workflow-init analyzes project and selects optimal track

    Document Sharding Strategy

    90% Token Savings Through Selective Loading

    BMad uses document sharding to dramatically reduce context overhead during implementation:

    • Planning Phase: Large documents (PRD: 50KB, Architecture: 80KB)
    • Sharding Process: Break into epic files (3-5KB) and story files (2-5KB)
    • Implementation Phase: Developer loads only single story (5KB vs 130KB)
    • Context Preservation: Each story embeds relevant epic and PRD sections
    • discover_inputs Protocol:
      • FULL_LOAD - Planning workflows load all documents
      • SELECTIVE_LOAD - Load only relevant epic
      • INDEX_GUIDED - Load only current story

    Key Differences

    SpecKit: Executable Spec Approach

    Philosophy: Your specification is the single source of truth that any AI assistant can execute against.

    • Commands: /specify, /plan, /tasks, /implement
    • Developer acts as workflow orchestrator
    • Works with existing AI tools (Copilot, Claude, Gemini)
    • Lower barrier to entry, minimal setup

    BMAD: Multi-Agent Team Approach

    Philosophy: A complete project team in a box with specialized AI agents collaborating through structured workflows.

    • Agents: Mary (Analyst), John (PM), Winston (Architect), Amelia (Developer), Sarah (SM), etc.
    • Workflows: *workflow-init, *create-prd, *create-architecture, *dev-story, etc.
    • Automatic track selection based on project complexity
    • Higher initial investment, greater control and scalability

    Example Scenario: Email Verification Feature

    With GitHub SpecKit:

    1. Run /specify ‚Üí AI generates detailed spec
    2. Run /plan ‚Üí AI creates technical plan (API, DB, stack)
    3. Run /tasks ‚Üí AI breaks into granular tasks
    4. Implement tasks with your preferred AI tool

    With BMAD METHOD:

    1. Run *workflow-init ‚Üí Analyzer selects Quick Flow track
    2. PM creates tech-spec.md with requirements
    3. Developer implements with *quick-dev
    4. Automated status tracking in sprint-status.yaml
